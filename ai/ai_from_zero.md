# AI 全链路：从打字到蹦字，每一步在算什么

## 写在前面

这份文档写给**只用过 AI 工具、但不知道背后发生了什么**的普通开发者。

你打开 Cursor 问「这段代码什么意思」，看到 AI 一个字一个字回复你——这就是你的全部认知。这份文档从这一点出发，**顺着一次请求的流动方向**，把路径上出现的每个概念当场讲清楚。不先背术语再碰计算——概念从计算过程中自然长出来。

**全文用同一句话贯穿所有技术例子：「我喜欢吃红烧肉」。** 你可以在每一节看到这 4 个 token 的向量如何一步步变形——从一串编号，到向量矩阵，到 Attention 运算，到 FFN 加工，到最终产出下一个 token。每一步都标明进来是几维、出去是几维、做了什么运算、这么算的实际意义是什么。

读完你会知道：

1. 你打的那行字，经历了什么变成了 AI 的回复
2. 每个环节在算什么、为什么这么算
3. 什么是 AI Infra，为什么这个工种存在

## 2. 分词：文本变数字

大模型不读汉字，只认数字。分词器（Tokenizer）做两件事，**先后进行**——先切 token，再查词表拿编号。位置信息在这一步**还不存在**。

```
"我喜欢吃红烧肉"
    ↓ 第 1 步：切分（按规则把文本拆成最小单位）
[我,  喜欢,  吃,  红烧肉]   ← 4 个 token（字符串）
    ↓ 第 2 步：查词表（每个 token 在词表里有一个固定编号）
[8247, 15302, 901, 43891]   ← 4 个词表编号（整数，训练时任意分配）
```

> **token 是什么？** 模型处理文本的最小单位。中文里可能是一个字（「我」）、一个词（「喜欢」）、一个标点（「。」）。token 不一定等于完整的一个字。收费按 token 数算，速度按 tokens/s 算，显存占用跟 token 序列长度直接相关。你看到的「蹦字」，底层是「蹦 token」。

> **容易误解：上面的数字是词表编号，不是位置编号。**
>
> | 编号类型 | 含义 | 本例 |
> |---|---|---|
> | **词表编号** | 这个 token 在词表里是几号 | 「我」→ 8247，「喜欢」→ 15302 |
> | **位置编号** | 这个 token 在句子里排第几位 | 「我」→ 位置 0，「喜欢」→ 位置 1 |
>
> 词表编号是训练时任意分配的，没有规律——8247 和 15302 之间不代表「差了多少语义」。分词器只产出词表编号。位置信息要到下一步 Embedding 才加进去（见第 3.1 节）。

实际代码里通常一行 `tokenizer.encode("我喜欢吃红烧肉")` 就直接返回 `[8247, 15302, 901, 43891]`，看起来像一步完成，但底层仍是：先切 token，再查表。

分词器跟模型是绑定的——训练时用什么分词器，推理时就必须用同一个，不能混用。GPT 和 Claude 切同一句话，token 数可能差 20%，直接影响计费和速度。

分词发生在**服务端**，不是在你的电脑上。你发出去的是文本，到机房后才被切成编号。

**现在，「我喜欢吃红烧肉」变成了一串词表编号 `[8247, 15302, 901, 43891]`。位置还没加。然后呢？**

## 3. 向量化 + 模型是什么

### 3.1 数字变向量（Embedding）

每个**词表编号**被映射成一个**向量**——一串有固定长度的浮点数。

**为什么要变向量，不直接用词表编号 8247、15302……？** 因为词表编号是任意分配的——901 和 8247 之间没有「差了多少语义」这种数学关系。向量里的每个数字是训练出来的，**语义接近的词，向量也会接近**——「猫」和「狗」的向量距离，比「猫」和「电脑」近。这个映射表（Embedding 表）是训练时学出来的。

向量的长度叫**维度 d**。

> 实际模型的 d 很大：Llama-3-8B 用 d=4096，Qwen3.6-35B-A3B 用 d=2048。**接下来的全部例子统一用 d=5**——原理完全一样，5 个数字你能看清楚，4096 个数字看不清楚。

4 个 token，每个映射成长度为 5 的向量（5 个浮点数）：

```
词表编号  →  5 维向量（5 个浮点数）

8247  ("我")     → [0.8,  0.1,  0.3, -0.2,  0.5]
15302 ("喜欢")   → [0.2,  0.9,  0.1,  0.4, -0.3]
901   ("吃")     → [0.1,  0.3,  0.8, -0.1,  0.6]
43891 ("红烧肉") → [0.3,  0.2,  0.4,  0.9, -0.1]
```

每个 token 现在是「5 个数字的列表」——这就是一个 **5 维数组**。此时向量里只有语义，**还没有位置信息**。

接着，模型给每个向量注入**位置信息**——根据它在句子里的顺序（第 0 位、第 1 位、第 2 位、第 3 位）。主流开源模型（Llama、Qwen 等）用 **RoPE（旋转位置编码）**：在 Q/K 投影后按位置旋转向量，而不是简单相加。下文为便于手算，仍用「位置向量直接相加」示意——**语义不变：让模型知道词序**。

```
位置 0（"我"）    的向量 + 位置0的编码 → 带位置的向量   ← 示意写法；Llama 系实际是 RoPE
位置 1（"喜欢"）  的向量 + 位置1的编码 → 带位置的向量
位置 2（"吃"）    的向量 + 位置2的编码 → 带位置的向量
位置 3（"红烧肉"）的向量 + 位置3的编码 → 带位置的向量
```

注入位置信息后：

- 向量里既有「这个词是什么意思」（语义）
- 也有「这个词在第几位」（位置）

**为什么需要位置编码？** 因为后面的 Attention 点积本身不区分顺序——没有位置信息的话，「我喜欢吃红烧肉」和「红烧肉吃喜欢我」对模型来说一模一样，词序就丢失了。

4 个 token、每个 5 维，按顺序从上到下摞起来，就是一个矩阵，记作 **X[4×5]**——4 行（4 个 token）× 5 列（d=5 维）：

```
         dim1  dim2  dim3  dim4  dim5
"我"    [ 0.8,  0.1,  0.3, -0.2,  0.5]   ← 第 1 行
"喜欢"  [ 0.2,  0.9,  0.1,  0.4, -0.3]   ← 第 2 行
"吃"    [ 0.1,  0.3,  0.8, -0.1,  0.6]   ← 第 3 行
"红烧肉"[ 0.3,  0.2,  0.4,  0.9, -0.1]   ← 第 4 行
```

**X[4×5]（4 个 token，每个 5 维）就是从这里进入模型的输入矩阵。** 接下来进入第 3.2–3.3 节说的那个「巨大数学函数」——权重已经训练好、装在 GPU 里。矩阵乘法不熟？先看第 3.3 节末尾的速成框，再读第 4 节 Attention。

### 3.2 模型是什么

模型就是一个巨大的数学函数：

```
输出 = f(输入, 权重)
```

拆开是两个东西：

| | 是什么 | 怎么来的 | 变不变 |
|---|---|---|---|
| **架构 f**（神经网络） | 运算流水线——多少层、每层做什么类型的运算 | 人设计的 | 固定 |
| **权重** | 流水线上每个操作的具体数字——乘几、加几 | 训练出来的 | 同一架构不同训练产出不同权重 |

**类比：** 菜谱步骤（先切再炒最后调味）= 架构。盐 3g、糖 5g = 权重。同一份菜谱，不同厨师练出来的用量不同，味道就不一样。

**神经网络**就是那个「运算流水线」——数据从一头进去，经过几十到上百「层」，每层做一批固定类型的数学运算（主要是矩阵乘法和加法），从另一头出来。

### 3.3 权重怎么来的（轻量版）

权重是**训练**出来的。训练程序做的事极其简单：**给定前面的字，猜下一个字是什么。**

```
输入："我喜欢吃"  → 模型猜 "红烧肉"  → 对比正确答案 → 猜错了
       ↓
反向传播：从最后一层往回，精确算出每个权重对猜错贡献了多少
       ↓
微调所有权重，往减小误差的方向拧一点点
       ↓
重复万亿次，直到模型猜下一个字很准
```

这就是**预训练（Pre-training）**——产出的是 Base Model，会接话但不会对话。后面还要经过 SFT（学对话格式）、偏好对齐（学风格）、推理增强（学思考）。详细训练流程见第 11 节。

**你的 4 个 token 的向量矩阵 X[4×5]，现在就进入这个模型。**

下面进入全文最核心的部分：**第 4 节 Attention**——让 token 之间互相"看到"彼此。然后**第 5 节 FFN**——对每个 token 独立做深度加工。这两块加在一起就是一个 Transformer 层，模型把它串几十遍，就是 Prefill 和 Decode。

> **矩阵乘法速成（30 秒）** — 读第 4 节之前建议先看这里
>
> 后面会频繁出现「A × B」。两条规则：
> 1. **第一个矩阵的列数，必须等于第二个矩阵的行数**（否则没法相乘）
> 2. **结果的形状 = 第一个矩阵的行数 × 第二个矩阵的列数**（中间那个维度"消掉了"）
>
> 用一个行列不等的例子看清楚：**[2行×3列] × [3行×2列] → [2行×2列]**
>
> ```
> [1  2  0]       [3  0]       [1×3+2×1+0×2,  1×0+2×4+0×5]   =   [5,   8]
> [0  1  3]  ×    [1  4]   =   [0×3+1×1+3×2,  0×0+1×4+3×5]       [7,  19]
>                 [2  5]
> ```
>
> 结果左上角 = **第1行** · **第1列** = 1×3 + 2×1 + 0×2 = **5**
> 结果右上角 = **第1行** · **第2列** = 1×0 + 2×4 + 0×5 = **8**
>
> 注意：两个矩阵中间那个维度（这里是 3）消掉了，结果只保留行数(2)×列数(2)。
>
> 「向量 × 矩阵」是同一回事：把长度为 5 的向量看成 [1行×5列]，乘一个 [5行×5列] 的矩阵 → 结果是 [1行×5列]，5 列各做一次「对应乘再求和」。
>
> 后面每次出现 `×`，就是在做这件事。

## 4. Attention：token 之间互相"看"

### 4.1 为什么需要 Attention

X[4×5] 进入模型，面对一个问题：每个 token 的向量目前只包含自己的孤立含义——「红烧肉」只知道自己是"红烧肉"，不知道前面有「喜欢」在修饰它，也不知道有「吃」在描述动作。

Attention 的目标：让每个 token **看到并加权利用其他 token 的信息**，把「红烧肉」的向量从「孤立的食物概念」更新成「有人喜欢吃的食物」的意思。

具体做法分两段：先用 X 乘三组权重矩阵（WQ、WK、WV），得到 Q、K、V；再用 Q、K、V 算出每个 token 该关注谁、借多少信息；最后把汇总结果乘第四组权重 WO，变换成主干网络能读的格式。六步细节见 4.2 节。

### 4.2 完整计算：六步走一遍

**全程用「我喜欢吃红烧肉」，d=5，4 个 token。从 X[4×5] 进，到 attn_out[4×5] 出。**

先看路线图——六步分别做什么、每步的输入输出形状怎么变：

```
第1步: X[4×5] × WQ/WK/WV → Q[4×5], K[4×5], V[4×5]   ← 把输入投影到三个"视角"
第2步: Q[4×5] × Kᵀ[5×4] ÷ 2.24 → scores[4×4]         ← 打分：每对 token 有多相关
第3步: 因果 mask（右上角 → -∞）                         ← 禁止偷看未来
第4步: softmax 每行 → weights[4×4]                     ← 分数变比例，每行和为1
第5步: weights[4×4] × V[4×5] → context[4×5]           ← 按比例借其他 token 的信息
第6步: context[4×5] × WO[5×5] → attn_out[4×5]         ← 变换回主干网络格式
最后:  attn_out + X（残差）→ hidden[4×5]               ← 原信息保留，只加增量
```

注意第 2→4 步形状从 [4×5] 变成 [4×4]——行列含义暂时从「token×维度」变成「token×token」。第 5 步之后变回来。下面逐步展开。

---

**第 1 步：三组投影 — X[4×5] × WQ/WK/WV[5×5] → Q[4×5], K[4×5], V[4×5]**

WQ、WK、WV 是三个独立的 5×5 权重矩阵（三张各有 25 个浮点数的表格，训练时学出来的）。**为什么是 5×5？因为 d=5**——每个 token 的向量是 5 维，权重矩阵的行数必须跟向量维度对齐才能相乘，列数决定投影出来还是几维。规律是：

```
X[token数 × d]  ×  WQ/WK/WV[d × d]  →  Q/K/V[token数 × d]
     [4×5]    ×       [5×5]         →      [4×5]
```

token 数量（4）只决定矩阵有几行，**不影响** WQ 是几×几——不管来 4 个 token 还是 4000 个，WQ 始终是 [d×d]。实际模型 d=4096 时，这三个矩阵就是 [4096×4096]。

用矩阵乘法把 X 投影到三个不同的空间：

```
Q = X[4×5] × WQ[5×5]  →  Q[4×5]
K = X[4×5] × WK[5×5]  →  K[4×5]
V = X[4×5] × WV[5×5]  →  V[4×5]
```

> 套用速成规则：X 是 [4行×5列]，WQ 是 [5行×5列]，中间的 5 消掉 → Q 是 [4行×5列]。
> 具体算一行：「我」是 X 的第 1 行（5 个数），乘 WQ 后得到 Q 的第 1 行（5 个数）：
> `Q("我")[1] = 0.8×wq₁₁ + 0.1×wq₂₁ + 0.3×wq₃₁ + (-0.2)×wq₄₁ + 0.5×wq₅₁`
> 4 个 token 同时算，一次矩阵乘法全部出来。

**这三个空间分别有什么实际含义？**

| | 角色 | 类比 |
|---|---|---|
| **Q（Query，查询）** | 「这个 token 在问什么、需要什么信息」 | 搜索框里打的词 |
| **K（Key，键）** | 「这个 token 能提供什么、有什么标签」 | 每篇文章的标题 |
| **V（Value，值）** | 「这个 token 的实际内容，传给别人的信息」 | 文章正文 |

**Q 和 K 为什么要分开？** 「我需要什么」和「我是什么」是两种信息视角。「红烧肉」作为 Q 时表达"需要找跟吃相关的信息"，作为 K 时表达"我是一道菜、是猪肉做的"。两个视角分开训练，模型能更精确地描述每个方向。

---

**第 2 步：打分 — Q[4×5] × Kᵀ[5×4] → scores[4×4]，再除以 sqrt(5)≈2.24**

```
          Q               Kᵀ              scores
   [4tokens × 5dims]  ×  [5dims × 4tokens]  →  [4tokens × 4tokens]
          ↑                    ↑                       ↑
    4 个 token         转置后变成                每对 token 的
    每个 5 维          5dims×4tokens              相关度分数
                       （中间的 5dims 消掉）
```

Kᵀ 是 K 的**转置**——把 K 的行和列互换：K 原本是 [4tokens×5dims]，转置后变成 [5dims×4tokens]。Q[4×5] × Kᵀ[5×4] 中间的 5dims 对齐相乘，结果 scores[4×4] 的含义变成了「4个token × 4个token」——每格是一对 token 之间的相关度分数。注意形状变化：输入 Q 和 K 都是 [4×5]（token×dim），输出 scores 是 [4×4]（token×token），行列含义彻底改变了。

**点积越大，说明两者方向越对齐——需求和供给越匹配。**

```
              K("我") K("喜欢") K("吃") K("红烧肉")
Q("我")     [  0.9     0.3      0.1      0.0    ]
Q("喜欢")   [  0.3     0.8      0.4      0.1    ]
Q("吃")     [  0.2     0.4      0.7      0.5    ]
Q("红烧肉") [  0.1     0.2      0.5      0.8    ]
```

然后每个数除以 `sqrt(d) = sqrt(5) ≈ 2.24`，保持数值稳定——维度越大点积越容易偏大，不除的话 softmax 后概率会极度集中到一个 token（几乎只有一个接近 1，其余接近 0），其他 token 的信息就基本浪费掉了。

> **实际意义：** 「吃」的 Q 跟「红烧肉」的 K 匹配分 0.5，比跟「我」的 0.2 高——训练后模型知道「吃」更需要关注被吃的对象。这是 WQ、WK 在海量文本上训练后自然产生的，不是人工规定的。

---

**第 3 步：因果掩码 — scores[4×4] → scores[4×4]（右上角变 -∞）**

```
加 mask 后：
              K("我") K("喜欢") K("吃") K("红烧肉")
Q("我")     [  0.9     -∞       -∞       -∞     ]  ← 只能看自己
Q("喜欢")   [  0.3     0.8      -∞       -∞     ]  ← 能看"我"和自己
Q("吃")     [  0.2     0.4      0.7      -∞     ]  ← 能看前三个
Q("红烧肉") [  0.1     0.2      0.5      0.8    ]  ← 能看全部
```

**为什么要掩掉未来？** 语言模型的训练任务是「根据前面猜下一个」。如果「我」能直接看到后面的「喜欢」，等于提前抄了答案，模型就不需要真正学习预测。把未来的分数设成 `-∞`，后续 softmax 会把它们变成 0，彻底禁止信息从未来流向过去。

注意：四个 token 的整张分数表是**一次矩阵乘法同时算出来的**，没有逐个 token 的循环。掩码把右上角设成 `-∞`（softmax 之后这些位置的概率才变成 0），并行计算不受影响。

---

**第 4 步：softmax — scores[4×4] → weights[4×4]**

每行做一次 softmax：每个数取 eˣ，再除以这一行所有 eˣ 的总和。

结果：**每行都是正数，且和为 1**——可以当作「注意力分配比例」来读。

```
weights（注意力权重）：
              "我"   "喜欢"  "吃"  "红烧肉"
"我"        [ 1.00   0.00   0.00   0.00 ]  ← 只看自己
"喜欢"      [ 0.38   0.62   0.00   0.00 ]  ← 38% 给"我"，62% 给自己
"吃"        [ 0.27   0.33   0.40   0.00 ]  ← 分散注意力
"红烧肉"    [ 0.20   0.22   0.27   0.31 ]  ← 均匀关注所有前文
```

> **实际意义：** 「红烧肉」那一行告诉我们，更新它的向量时，分别从「我」借 20%、「喜欢」借 22%、「吃」借 27%、自身 31%。这个分布是训练后自然产生的，不是人工规定的。

---

**第 5 步：加权汇总 V — weights[4×4] × V[4×5] → context[4×5]**

```
          weights          V             context
   [4tokens × 4tokens]  ×  [4tokens × 5dims]  →  [4tokens × 5dims]
                            （中间的 4tokens 消掉）
```

每个 token 的 context 向量 = 按注意力权重，对所有 token 的 V 加权求和：

```
context("红烧肉") = 0.20 × V("我") + 0.22 × V("喜欢") + 0.27 × V("吃") + 0.31 × V("红烧肉")
context("我")     = 1.00 × V("我")   ← 只能用自己的 V
```

> **理解要点：同一套注意力权重作用在 V 的所有维度上。** 矩阵乘法中，weights 的每一行跟 V 的每一列做点积——"红烧肉"那行的 `[0.20, 0.22, 0.27, 0.31]` 对 V 的 dim1、dim2...dim5 各算一次，同一套权重一视同仁。不存在"dim1 借 20% 但 dim2 借 50%"——对不同维度分配不同权重是后面 FFN 的事，不是 Attention 的事。

V 每行是 5 维，加权求和后每行还是 5 维。形状从 weights[4×4] 和 V[4×5] 变成了 context[4×5]——行列含义又变回了「token × dim」。

> **实际意义：** context 里每个 token 的向量，从「孤立的词义」变成了「带上下文的词义」。「红烧肉」从泛指食物，变成了「被喜欢吃的食物」。

---

**第 6 步：输出投影 — context[4×5] × WO[5×5] → attn_out[4×5]**

```
attn_out = context[4×5] × WO[5×5]  →  attn_out[4×5]
```

WO 是 5×5 的权重矩阵，跟 WQ/WK/WV 一样是训练学出来的，存在模型包的权重文件里。

**为什么还需要 WO？** context 里的信息是 V 向量的加权混合，V 存在自己的"子空间"（V 的各个维度方向），跟整个网络主干期望的方向不一定对齐。WO 的作用是做一次线性变换，把 context 重新表达成「主干网络能读懂」的格式——后续的 FFN 和下一层 Attention 才能正确读取。

类比：你汇总了 4 个人的笔记（context），但下一道工序的人按不同的维度理解信息——WO 负责把你的汇总结果重新编排成他能懂的格式。

> **👆 以上六步就是 Attention 的完整计算。下面讲多头和 GQA——初读可跳过，不影响理解主线。读完 Prefill/Decode 后再回来看。**
>
> **多头 Attention（Multi-Head Attention）**：上面展示的是单头简化版。实际模型在**同一层内**并行运行多套 WQ/WK/WV（每套叫一个「头」），每个头在更低的维度上独立做 Attention，最后拼接起来乘 WO。
>
> **为什么要多头？** 单头时所有维度揉在一起做点积，只能学到一种「关注模式」。拆成多个头（如 8 头、d=512，则每头 d_head=64），各头在 64 维的低维子空间里独立计算——互不干扰，不同头自然学会关注不同东西（一个偏句法、一个偏指代、一个偏情感）。
>
> **两层结构：**
> - **层与层之间**：权重完全不共享（第 1 层和第 2 层各有自己的 WQ/WK/WV/WO）
> - **同一层内部**：又拆成多个头并行计算
>
> **GQA（Grouped-Query Attention，分组查询注意力）**：Llama 3 等主流模型在多头之上还有一层优化——Q 仍有 32 个头，但 K/V 只保留 8 个头，多组 Q 共享同一组 K/V。推理时 KV Cache 按 K/V 头数存，能省约 4 倍显存，计算前会把 K/V 复制到与 Q 头数对齐。
>
> **每个头的计算（以 d=512、8 头为例）：**
> ```
> d_head = 512 / 8 = 64
>
> Q_i = X[4×512] × WQ_i[512×64]  →  [4×64]
> K_i = X[4×512] × WK_i[512×64]  →  [4×64]
> V_i = X[4×512] × WV_i[512×64]  →  [4×64]
>
> scores_i → softmax → × V_i  →  head_i[4×64]
> ```
> 8 个头拼接：`concat(head₁...head₈) → [4×512]`，再乘 WO[512×512] 融合各头信息。WO 在多头下除了变换空间，还负责融合各头——哪个头重要、怎么混合，都是训练学出来的。

---

**残差连接：attn_out[4×5] + X[4×5] → hidden[4×5]**

```
hidden = attn_out + X
```

把 Attention 的输出和 Attention 的输入直接相加。这叫**残差连接（Residual Connection）**。

**意义：** Attention 只需要学「需要在原有基础上修改什么」，而不需要学「从头重写整个向量」——原始信息通过加法原路保留。这在网络很深时（几十层）至关重要。反向传播调整权重时，梯度要从最后一层逐层往回传。没有残差连接，每传一层梯度就衰减一次，传到前几层时已经接近于零——那些层的权重就几乎不会被更新，等于白堆了。残差连接给梯度提供了一条不经过矩阵乘法的直通路径，让每一层都能收到足够强的调整信号。

---

**Attention 全程形状汇总：**

```
X[4×5]（进来：4个token，每个5维）
  ↓ × WQ/WK/WV[5×5]
Q[4×5]  K[4×5]  V[4×5]       ← 还是 token×dim
  ↓ Q[4×5] × Kᵀ[5×4] ÷ 2.24
scores[4×4]（token×token，含义变了！）
  ↓ 因果 mask（右上角 → -∞）
scores[4×4]（未来位置被屏蔽）
  ↓ softmax（每行）
weights[4×4]（每行和为 1，可读作注意力比例）
  ↓ × V[4×5]
context[4×5]（token×dim，含义变回来）
  ↓ × WO[5×5]
attn_out[4×5]（变换回主干网络空间）
  ↓ + X[4×5]（残差）
hidden[4×5]（出去，传给 FFN）
```

**进来 X[4×5]，出去 hidden[4×5]——形状不变，但每个向量已经融合了上下文信息。**

---

#### 补充：位置编码的作用

第 3.1 节注入了位置信息（下文用「相加」示意；Llama 系实际是 RoPE 作用在 Q/K 上）。位置信息在 Attention 打分里发挥作用：Q 和 K 都携带「我是第几位」的信号，Q×Kᵀ 算出来的分数，既反映「语义相关性」，也反映「位置关系」。因果 mask 决定了信息只能从左往右流：

```
「我吃了红烧肉」   → 到「红烧肉」时，已经知道主语"我"和动作"吃"
「红烧肉被我吃了」 → 到「红烧肉」时，还不知道后面会有"被吃"这个信息
```

同一个词在不同语序下，Attention 融合的上文不同，输出向量自然不同。

> **为什么位置编码只加在 Q/K，不加在 V？** 因为位置的作用在打分阶段（Q×Kᵀ）就已经完成了——分数已经包含了「谁离谁近」的信息。V 是被动的内容载体，加权求和时直接用这些分数做比例，位置不需要再参与。加在 V 上反而会让内容向量被位置信息"污染"。这也是为什么 RoPE 的原始论文只对 Q/K 做旋转——V 不参与打分。

---

## 5. 一层不够：Attention + FFN = 一个 Transformer 层

### 5.1 FFN 是什么，做什么

Attention 让 token 之间通信，得到 hidden[4×5]——每个 token 的向量已经融合了上下文。但这些信息只是被**线性地加权混合**了（加权平均），还没有被深度加工。

FFN（Feed-Forward Network，前馈网络）的目标：**对每个 token 的向量做深度加工**——先膨胀到更大维度（给模型足够空间表达复杂模式），用非线性激活做筛选，再压缩回原来的维度。

> **FFN 存着模型的"知识"。** 研究者发现，模型里大量的事实性知识（「巴黎是法国首都」「红烧肉的主料是五花肉」）主要存储在 FFN 的权重里——Attention 负责在 token 间传递信号，FFN 负责在自己的权重里"回忆"和"运用"这些知识。

**统一设定：d=5，FFN 中间维度 = 10（约 2 倍膨胀，只为手算好看；Llama-3-8B 实际约 3.5 倍，原理一样）。**

FFN 对 4 个 token **各自独立加工**——「红烧肉」的 FFN 计算不会影响「我」的 FFN 结果，token 之间没有信息交叉（这是 Attention 的事）。以「红烧肉」为例：

**起点：「红烧肉」经 Attention 后的向量 = 5 维数组**

```
hidden("红烧肉") = [0.5,  0.3, -0.2,  0.8,  0.4]   ← 5 个数字（5 维）
```

SwiGLU FFN 对**同一份 5 维输入**走两条并行路径，再汇合：

---

**第 1 步：门控路径 — 数组[5] × W_gate[5×10] → silu → gate[10]**

W_gate（Llama 源码里叫 `gate_proj`）是一个 5 行、10 列的权重矩阵（5×10=50 个浮点数，训练时学出来的）。

```
输入:  [0.5,  0.3, -0.2,  0.8,  0.4]            ← 5 维
         ↓  × W_gate[5×10]
gate_raw: [1.2, -0.4, 0.8, 0.1, -0.9, 1.5, 0.3, -0.6, 0.7, 0.2]   ← 10 维
         ↓  silu（逐元素，见下方说明）
gate:     [0.9, -0.2, 0.6, 0.1, -0.2, 1.3, 0.4, -0.2, 0.5, 0.3]   ← 10 维
```

**5 维 → 10 维。** gate 这一路负责「每个中间维度该放行多少」。

> **silu 是什么？** `silu(x) = x × sigmoid(x)`，也叫 Swish。比 ReLU 平滑：正值大致保留，**负值被衰减而不是直接归零**——所以 gate 里仍可能带着少量负向信号，细节不会一刀切丢掉。

---

**第 2 步：值路径 — 数组[5] × W_up[5×10] → up[10]**

W_up（`up_proj`）是另一套独立的 5×10 矩阵，跟 W_gate **形状相同、权重不同**。输入是同一份 hidden，但投影到不同的 10 维空间：

```
输入:  [0.5,  0.3, -0.2,  0.8,  0.4]            ← 还是那份 5 维
         ↓  × W_up[5×10]
up:    [0.8,  0.5, -0.3, 0.6,  0.2,  1.1, -0.4, 0.7,  0.3,  0.9]   ← 10 维
```

**5 维 → 10 维。** up 这一路负责「每个中间维度实际携带什么内容」。

> **实际意义：** 膨胀靠 gate 和 up **两路各自独立投影**到 10 维——同一份 5 维输入，用两套不同权重展开到两个 10 维空间，再逐元素汇合。比老 GPT 单路径 GELU 多一套投影，表达能力更强。

---

**第 3 步：门控汇合 — gate[10] ⊙ up[10] → mid[10]**

⊙ 表示**逐元素相乘**（两个等长数组对位相乘），形状不变：

```
gate: [0.9, -0.2, 0.6, 0.1, -0.2, 1.3, 0.4, -0.2, 0.5, 0.3]
up:   [0.8,  0.5, -0.3, 0.6,  0.2,  1.1, -0.4, 0.7,  0.3,  0.9]
        ↓  ⊙（逐元素）
mid:  [0.7, -0.1, -0.2, 0.1, -0.0, 1.4, -0.2, -0.1, 0.2, 0.3]   ← 还是 10 维
```

这就是 SwiGLU 的核心：**silu(gate路径) × up路径**，用 gate 决定每个维度放行 up 里的多少内容。负的 gate 值会压低对应维度，但不一定会把它清零。

> **实际意义：** 这是整个 FFN 里**引入非线性的关键步**。如果没有 silu 和逐元素乘，连续多层矩阵乘数学上可合并成一层，深度就失去意义。SwiGLU 让每一层能表达「如果…则…」的条件逻辑。

---

**第 4 步：压缩 — mid[10] × W_down[10×5] → out[5]**

W_down（`down_proj`）是一个 10 行、5 列的权重矩阵（10×5=50 个浮点数）：

```
输入:  [0.7, -0.1, -0.2, 0.1, -0.0, 1.4, -0.2, -0.1, 0.2, 0.3]   ← 10 维
         ↓  × W_down[10×5]
输出:  [0.7, -0.3, 1.1, 0.5, 0.2]                               ← 5 维
```

**10 维 → 5 维，维度压缩回来。**

> **实际意义：** 把在 10 维空间里门控加工过的信息压回 5 维，与模型主干维度一致——下一层 Attention 期待的输入也是 5 维。W_down 在训练中学会「哪些中间维度的信息最有价值，优先保留」。

（Llama / Qwen / DeepSeek 的 FFN **到此结束，没有 b₂**；老 GPT 还会在压缩后加偏置 b₂。）

---

**残差连接：FFN 输出[5] + 进入 FFN 前的 hidden[5] → 新 hidden[5]**

```
新 hidden("红烧肉") = FFN 输出 + 进入 FFN 之前的向量
```

和 Attention 的残差连接一样——FFN 只需要学「需要额外添加什么」，原始信息原路保留，训练更稳定。

> **SwiGLU 是当前主流，但不是唯一方案。** Llama 3、Qwen、DeepSeek 等开源模型用 SwiGLU（三套投影：`gate_proj` / `up_proj` / `down_proj`，Llama 系不带偏置 `bias=False`）。更早的 GPT-2/GPT-3 用单路径 GELU（一次膨胀→激活→压缩，带偏置 b₁/b₂）；原始 Transformer 用 ReLU。Llama-3-8B 的 FFN 中间维约 14336，相对 d=4096 约 3.5 倍（SwiGLU 三矩阵换参后，与经典 4× 膨胀大致同量级）。

---

**FFN 全程形状汇总（单个 token，SwiGLU）：**

```
hidden[5]（来自 Attention）
  ├─ × W_gate[5×10] → gate_raw[10] → silu → gate[10]   ← 门控路径
  └─ × W_up[5×10]   → up[10]                            ← 值路径（与 gate 并行）
gate[10] ⊙ up[10] → mid[10]                              ← 门控汇合（非线性在这里）
  ↓ × W_down[10×5]
out[5]（压缩回 5 维）
  ↓ + hidden[5]（残差）
新 hidden[5]（传下一层）
```

**5→10→10→10→5→5。进去 5 维，出来 5 维。**

对 4 个 token 批量处理时：输入 [4×5]，W_gate / W_up 各是 [5×10]，分别乘出 [4×10]；逐元素乘后仍 [4×10]；W_down [10×5] 乘完 [4×5]。4 个 token 并行算，结果互不影响。

---

**一个完整 Transformer 层的形状汇总：**

```
输入 X[4×5]（4 个 token，每个 5 维）
──── Attention 块 ────
  × WQ/WK/WV[5×5]   →  Q[4×5], K[4×5], V[4×5]
  Q[4×5] × Kᵀ[5×4] ÷ 2.24  →  scores[4×4]   ← 含义变：token×dim → token×token
  因果 mask           →  scores[4×4]（右上角 -∞）
  softmax（每行）     →  weights[4×4]
  × V[4×5]            →  context[4×5]           ← 含义变回：token×token → token×dim
  × WO[5×5]           →  attn_out[4×5]
  + X（残差）         →  hidden[4×5]
──── FFN 块（SwiGLU）────
  × W_gate[5×10]      →  gate_raw[4×10] → silu → gate[4×10]   ← 门控路径
  × W_up[5×10]        →  up[4×10]                              ← 值路径（并行）
  gate ⊙ up           →  mid[4×10]                             ← 门控汇合
  × W_down[10×5]      →  out[4×5]               ← 压缩：10维 → 5维
  + hidden（残差）    →  新 X[4×5]
输出 新 X[4×5]（传给下一层）
```

### 5.2 MoE 替换的是 FFN

标准 FFN 的 W_gate / W_up / W_down 对所有 token 一视同仁——「红烧肉」乘的矩阵和「我」乘的是同一套。MoE（Mixture of Experts，混合专家）的思路：不做一份巨大的 FFN，而是准备多份较小的"专家"权重，让路由器根据 token 的内容决定分配给哪几个专家。

```
标准 FFN：                              MoE FFN：
每个 token 过同一套 gate/up/down       每个 token 先经过路由器
                                       ↓
                                      "红烧肉？分给专家3、7、15"
                                       ↓
                                      只有被选中的专家做计算
                                       ↓
                                      各专家输出加权合并
```

**关键效果：** 总参数量可以极大（384 个路由专家各有各的权重），但每个 token 只激活其中 6 个，实际计算量很小。DeepSeek V4-Pro 总参数 1.6T，每次只激活 49B（约 3%）。

**MoE 和 Attention 不冲突——** MoE 换的是 FFN 那块，Attention 照常运行。

### 5.3 多层堆叠：从字面到意图

一层 Attention+FFN 不够。实际模型堆几十到上百层（Llama-3-8B 有 32 层），每层在前一层的基础上继续加工。研究者用**探针实验**（在每层输出的向量上训练分类器，看能识别出什么信息）发现不同深度的层编码了不同层次的信息：

> 「小明说他昨天去了那家新开的川菜馆，菜很辣但是他很喜欢」
>
> 追踪最后一个 token「喜欢」在各层编码了什么：

```
第 1-4 层（表层编码）：
  向量里能检测到："喜欢"前面紧挨着"很"，再往前是"他"
  检测不到：这个"他"是谁、整句结构如何

第 5-12 层（句法编码）：
  能检测到：句子由两大块；"但是"标记转折；"他"与"小明"共指
  往前推进了：不再是相邻字关系，而是识别出句子骨架

第 13-24 层（语义编码）：
  能检测到："辣"和"喜欢"被"但是"隔开、情绪方向相反——语义冲突
  往前推进了：不只是谁修饰谁，开始编码语义矛盾

第 25-32 层（语用编码）：
  能检测到：矛盾被消解——"喜欢"是重心，"辣"被重新编码为正面信号
  往前推进了：整体情绪是正面的，意图是推荐而非抱怨
```

> **诚实说明：** 各层输出的仍然是向量（一串数字），不是自然语言。「表层/句法/语义/语用」这些描述，是研究者用探针实验对向量编码的信息做事后解读出来的——**不代表模型内部存在「句法模块」「语义模块」这样的独立零件**。开发者容易按「模块化架构」去想象，但这里没有那样的模块，整个网络就是一堆矩阵乘法连续堆叠，各层渐进地丰富向量里携带的信息，边界不是一刀切的。

## 6. Prefill：输入一次过完所有层

上面讲清楚了**一层** Transformer 内部在算什么。现在把它串起来——你的 4 个 token 如何一次过完所有层，产出第一个输出字。

分词器把用户输入变成 token 列表，经过 Embedding 变成向量矩阵 X 后，模型进入 **Prefill 阶段**。

**一句话：输入 token 列表 → 第 1 层 Attention+FFN → 第 2 层 Attention+FFN → ... → 第 L 层 → 末尾采样 → 第 1 个输出 token。** 模型有多少层，就串多少轮。

下面用「我喜欢吃红烧肉」4 个 token、模型 3 层来走一遍（实际模型有几十上百层，逻辑一样，3 层够看清结构）：

```
Prefill 开始
  输入: X₀[4×5]（4 个 token，每个 5 维）
  KV Cache: 空
       ↓
第 1 层
  Attention:
    Q,K,V = X₀[4×5] × WQ/WK/WV[5×5]    → 各 [4×5]
    scores = Q[4×5] × Kᵀ[5×4] ÷ 2.24   → [4×4]
    mask → softmax → × V[4×5]           → context[4×5]
    × WO[5×5]                            → attn_out[4×5]
    + X₀（残差）                         → hidden[4×5]
  💾 存 KV Cache[层0]: K[4×5], V[4×5]（这层算出来的）
  FFN:
    gate=silu(hidden×W_gate), up=hidden×W_up, mid=gate⊙up → out=mid×W_down → X₁[4×5]
       ↓
第 2 层
  Attention: X₁[4×5] → Q,K,V[4×5]（用 X₁ 算，不是重算 X₀）→ ...
  💾 存 KV Cache[层1]: K[4×5], V[4×5]
  FFN → X₂[4×5]
       ↓
第 3 层
  Attention: X₂[4×5] → ...
  💾 存 KV Cache[层2]: K[4×5], V[4×5]
  FFN → X₃[4×5]
       ↓
末尾采样
  取最后一个位置（"红烧肉"）的向量 X₃[行4] = 长度为 5 的数组
  → × lm_head[5×100000]（一个 5×10万 的权重矩阵）→ 100000 个分数
  → softmax → 100000 个概率（全正，和为 1）
  → 按策略挑一个 → 第 1 个输出 token，比如 "真"
```

> **为什么只取最后一个位置？** Prefill 时 4 个位置都会算完 Attention+FFN（训练时每个位置都学「猜下一个字」），但**推理接龙**只需要「整句输入之后，下一个字是什么」——所以只对最后一个位置做 lm_head + 采样。前面 3 个位置的输出向量在推理里不用于出字，但它们的 K/V 已经进了 Cache，供 Decode 复用。

**Prefill 结束。产物：** 第 1 个输出 token（"真"）；KV Cache = 3 层 × 4 行的 K 和 V。

**KV Cache 只存 K 和 V，不存每层的最终输出 X。** 存 K/V 是为了 Decode 阶段复用——后续每生成一个新 token，历史 token 的 K 和 V 不用重算，直接从 Cache 读出来拼接。

> **为什么只存 K/V，不存 Q？** Decode 每步只算 1 个新 token——只有这个新 token 需要问"我应该关注历史里的谁"（产生 Q）。历史 token 的 Q 在上一步已经用过了（它们当时问过"我应该关注谁"），后面不会再被问到。而历史 token 的 K 和 V 每次都要被新 token 的 Q 查询——所以 K/V 必须缓存，Q 不用。

**关键点：** Attention 和 FFN 在计算上都是矩阵并行——所有 token 堆成矩阵，一次矩阵乘全出，没有 `for token in tokens` 这种串行循环。Prefill 是完整 forward，只产出 1 个 token。

### KV Cache 到底占多大

**为什么作为一个开发者需要关心这个？** 因为 KV Cache 是长对话和多人同时用时的主要瓶颈——显存被 Cache 撑满，新的请求就进不来了。下面用真实数字感受一下（**简化估算**：按全维度 d 存 K/V；Llama 3 等用 GQA 时，K/V 头数更少，实际占用约为同 d 下的 1/4～1/8）：

```
输入 token 数 N = 1000
层数 L = 50
每层 KV 维度 d_kv = 1024   ← 简化：GQA 模型应换成 n_kv_heads × head_dim

KV Cache = L × N × d_kv × 2(K+V) × 2字节(FP16)
         = 50 × 1000 × 1024 × 2 × 2
         ≈ 200 MB（一个请求，按上式粗算）
```

100 个用户同时来 ≈ 20 GB，光 KV Cache，不算权重。

## 7. Decode：一个字一个字蹦

Prefill 产出了第 1 个输出 token（"真"）和全量 KV Cache（3 层 × 4 行）。之后进入 **Decode 循环**。

**Decode 做的事：拿上一个产出的 token（只是一个编号），走一遍完整的 Embedding + L 层 Attention+FFN + 末尾采样，产出下一个 token。**

与 Prefill 的关键区别：

| | Prefill | Decode（每一步） |
|---|---|---|
| **输入形状** | 矩阵 [4×5]（4 个 token 同时进） | 向量 [1×5]（1 个新 token） |
| **K/V 来源** | 当场从 4 个 token 全量计算 | 新 token 自己算 K/V，历史从 Cache 读 |
| **因果 mask** | 需要（4 个 token 有前后关系） | 不需要（每步只算 1 个新 token，它天然在序列最末，不会「偷看未来」） |
| **并行度** | 高——4 个 token 同时算 | 低——只有 1 个 token |
| **瓶颈** | 计算密集型：时间花在矩阵乘法上 | 内存密集型：每步读全部 KV Cache，时间花在搬运数据上 |
| **产出** | 第 1 个输出 token | 每步 1 个输出 token |

### Decode 第 1 步：从 "真" 到 "好"

Prefill 产出的 "真" 只是一个词表编号（比如 21056），先过 Embedding：

```
21056（"真"）→ Embedding 查表 → 长度为 5 的数组 X_new[1×5]
```

然后进第 1 层：

```
输入: X_new[1×5]（只有 1 个 token，不是 Prefill 的 [4×5]）

Attention:
  Q = X_new[1×5] × WQ[5×5]  →  Q[1×5]    ← 只有 1 行
  K = X_new[1×5] × WK[5×5]  →  K[1×5]
  V = X_new[1×5] × WV[5×5]  →  V[1×5]

  💾 KV Cache 操作:
     读 Cache[层0]: Prefill 存的 K[4×5], V[4×5]
     拼接: K_full = concat(历史K[4×5], 新K[1×5])  →  K_full[5×5]
           V_full = concat(历史V[4×5], 新V[1×5])  →  V_full[5×5]
     存回 Cache[层0]（现在 5 行，下次 Decode 会读 5 行）

  scores = Q[1×5] × K_fullᵀ[5×5]  →  scores[1×5]
  ↑ 1 个新 token 的 Q[1×5]，跟全部 5 个 token 的 Kᵀ[5×5] 打分，得 5 个分数
	（这步 K_full 恰好是方阵 [5×5]，转置后形状标记不变；下一步变 [6×5]，转置后就是 [5×6] 了）
  不需要 mask——新 token 是最后一个，后面没人需要屏蔽

  weights = softmax(scores[1×5])        →  weights[1×5]
  context = weights[1×5] × V_full[5×5]  →  context[1×5]
  attn_out = context[1×5] × WO[5×5]     →  attn_out[1×5]
  + X_new（残差）                        →  hidden[1×5]

FFN: gate=silu(hidden×W_gate), up=hidden×W_up, mid=gate⊙up, out=mid×W_down → [1×5]  （和 Prefill 完全相同的权重）

层输出: [1×5] → 传第 2 层
```

**第 2 层、第 3 层：** 完全相同的流程，各自读自己的 Cache，把 4 行扩到 5 行，用 5 行的 K/V 做 Attention，FFN，输出 [1×5] 往下传。

```
末尾采样:
  最后一层输出 [1×5]
  × lm_head[5×100000]  →  [1×100000] logits
  → softmax → 概率分布
  → 采样 → 第 2 个输出 token，比如 "好"
```

### Decode 第 2 步及之后

```
"好" 的编号 → Embedding → [1×5]
每层 Cache 从 5 行 → 6 行
... 同样的 L 层 Attention+FFN + 采样
产出: 第 3 个输出 token

重复直到产出结束 token（<eos>）或达到长度上限
```

### 为什么输出必须串行

第 N 个 token 依赖第 N-1 个 token——不知道上一个字是什么就算不出下一个。这叫**自回归**（Autoregressive），本质是超大号 Autocomplete：每次根据已有内容猜下一个最可能的字。

## 8. 采样：分数变文字

无论 Prefill 还是 Decode，所有层走完后，拿到最后一个位置的向量（长度为 d 的数组）。接下来把它变成一个具体的 token。

### 8.1 lm_head：从 d 维向量到词表分数

**lm_head 是什么？** 一个训练学出来的权重矩阵，存在模型包的权重文件里，推理时 load 进 GPU。跟 WQ、W_gate 一样是权重，专门负责最后这一步。

**形状：** `[d × vocab_size]`。d 行（跟模型维度一致），vocab_size 列（每列对应词表里的一个候选 token）。d 和 vocab_size 都写在 config.json 里。Llama 3 的 lm_head 与 Embedding **不共享**权重（`tie_word_embeddings=false`）。

```
最后一个位置的向量 [1×5] × lm_head[5×100000] = [1×100000]

出来 100000 个分数，每个对应词表里一个 token
```

**词表跟 lm_head 不是一个东西。** 词表是 tokenizer 里的查表规则——编号 1234 对应汉字"这"，存的是映射关系。lm_head 是权重矩阵，存的是训练学出来的浮点数。

### 8.2 softmax：把分数变成概率

lm_head 产出的 100000 个分数（叫 logits），可以是任意实数——正的负的大的小的。需要变成概率才能采样。softmax 做的事：

```
输入: [3.2,  1.5,  -0.8,  0.1]     ← 任意实数，有正有负

第1步: 每个数做 eˣ（指数函数）
       e^3.2 ≈ 24.5
       e^1.5 ≈ 4.5
       e^-0.8 ≈ 0.45               ← 负数不会消失，只是变小
       e^0.1 ≈ 1.1

第2步: 求和  24.5 + 4.5 + 0.45 + 1.1 = 30.55

第3步: 每个除以总和
       24.5 / 30.55 ≈ 0.80
       4.5 / 30.55 ≈ 0.15
       0.45 / 30.55 ≈ 0.01
       1.1 / 30.55 ≈ 0.04

输出: [0.80, 0.15, 0.01, 0.04]     ← 全正、和为 1，就是概率分布
```

> 回头看 Attention 第 4 步——那里也是 softmax，但输入里有 `-∞`（mask 加的）。`e^(-∞) = 0`，所以未来位置的概率变成 0。正常 softmax 没有 `-∞` 输入，负数会变成小的正概率，不会消失。

### 8.3 从概率分布里挑一个

100000 个候选，每个有一个概率。怎么挑：

- **贪心（Greedy）：** 永远选概率最高的——稳定但死板，容易重复
- **Temperature：** 采样前先把分数除以温度值。温度 > 1 → 概率更均匀 → 更随机（更有创意）。温度 < 1 → 概率更集中 → 更保守
- **Top-p：** 按概率从高到低累加，只在前 p% 的候选中随机挑——折中方案

挑出来的结果是一个 token 编号，通过**分词器查表**还原成可读文字（Detokenize）。

## 9. 流式返回：你看到字蹦出来

Decode 每产出一个 token，就通过 SSE（Server-Sent Events，HTTP 长连接）推送给客户端。客户端收到一段渲染一段——所以你看到「一个字一个字蹦出来」。

## 10. 全景回顾

### 完整链路

```text
你打字 → 客户端拼 JSON → HTTPS → 网关(鉴权/限流)
  → 路由(选模型/机器) → 分词(文本→token编号)
  → Embedding(编号→向量矩阵X[N×d])
  → Prefill(X过L层Attention+FFN，建KV Cache，产第1个token)
  → Decode循环(每次1 token，Cache追加，采样) → 流式推回
  → Detokenize(token编号→文字) → 你看到字蹦出来
```

### AI Infra 边界

```
❌ 不是 AI Infra: 客户端UI / 拼Prompt / 鉴权限流计费
✅ 是 AI Infra:   分词 → 推理引擎+框架 → KV Cache管理 → GPU调度 → 采样 → 流式推送
```

### GPU 显存里有什么

| 占用项 | 大小趋势 | 说明 |
|---|---|---|
| **权重** | 固定，最大 | 7B 模型约 14GB(FP16)，70B 约 140GB |
| **KV Cache** | 随对话增长 | 每多 1 token，各层多存一行 K/V；长对话 + 高并发 = 显存杀手 |
| **临时缓冲区** | 单次 forward | 算完释放 |
| **token 编号** | 极小 | 可忽略 |

> **本文有意简化的点（读完全文可回头对照）：** ① 位置编码用「相加」示意，Llama/Qwen 实际是 RoPE；② Attention 用单头 d=5，未展开多头与 GQA 的矩阵形状；③ FFN 中间维取 10 仅为手算，非 Llama 的 3.5×；④ KV Cache 估算按全维 d，GQA 模型实际更小；⑤ 未写 RMSNorm、LayerNorm 等归一化层（每层 Attention/FFN 前都有，不改变「矩阵进、矩阵出」的主线）。

### 为什么用 GPU 不用 CPU

GPU 有几千个计算核心，擅长同时做大量矩阵乘法——这正是 Attention 和 FFN 需要的。CPU 只有几十个核心。更关键的是：GPU 显存带宽极高（TB/s 级），权重必须常驻显存——如果每次 forward 从 CPU 内存通过 PCIe 总线搬运到 GPU，整体会慢几个数量级。

> **FP16 / 量化：** FP32 每个数占 4 字节，FP16 占 2 字节，INT8 占 1 字节。把 FP32 权重转成低精度存储叫**量化**——能省一半甚至更多显存，代价是轻微精度损失。

## 11. 训练：权重到底怎么练出来的

前面第 3.3 节给了轻量版。这里展开——一个大模型从零到能对话，经过四个阶段。

### 11.1 预训练（Pre-training）

**任务：** 海量文本上猜下一个字。

```
随机抽一段文本 → 给前面 N 个字让模型猜第 N+1 个字
  → 对比正确答案 → 猜错了
  → 反向传播：从最后一层往回，精确算出每个权重对猜错贡献了多少
  → 微调所有权重（贡献大的多拧，贡献小的少拧）
  → 重复万亿次
```

**数据：** 《三体》全文、维基百科、Stack Overflow、GitHub 公开代码、新闻网页……数万亿 token。这些文本**不进模型文件**——训练跑完后，模型文件里只有训练产出的数字（权重）。训练数据留在算法团队的集群里。

**产出：** Base Model——会接话，但不会对话。你问「这段代码什么意思」，它可能继续往下写代码而不是解释，因为它只学会了「猜下一个字」，还没学会「回答问题的格式」。

**成本：** 几万亿 token、成百上千张 GPU、几周到几个月、几百万到几千万美元。

### 11.2 SFT（监督微调，Supervised Fine-Tuning）

用人工写的「问题→标准答案」对子（通常 1 千到 5 万条）继续训练，让模型学会对话格式和遵循指令。

```
输入: "请解释这段代码的功能"
目标输出: "这段代码实现了一个二分查找算法，时间复杂度 O(logn)..."
```

这一步后模型学会了「用户问问题→我回答」的格式。但风格可能还很生硬。

### 11.3 偏好对齐（RLHF / DPO）

同一个问题给多个回答，人类标注哪个更好。模型学习「为什么 A 优于 B」——不再是简单对错判断。DPO（Direct Preference Optimization，直接偏好优化）是近年常见做法：可跳过单独训练奖励模型，直接从偏好数据里学，计算量比经典 RLHF 更小。

### 11.4 推理增强（GRPO / RLVR）

DeepSeek-R1 用的方法。用数学题答案、代码运行结果等**可验证的奖励信号**，让模型在试错中自己学会推理——没人逐步教它「怎么思考」，强化学习过程中可能涌现出更长的中间推理链（业界有时叫「aha moment」，但仍是研究解读，不是模型里真有个「顿悟模块」）。

### 11.5 训练阶段一览

| 阶段 | 做什么 | 数据规模 | 是「错了往对调」吗 |
|------|--------|----------|:---:|
| **预训练** | 猜下一个字，猜错→反向传播微调 | 数万亿 token | ✅ |
| **SFT** | 人工标注问答对，模仿标准答案 | 1K~50K 条 | ✅ |
| **偏好对齐** | 人类偏好排序，学"哪个答案更好" | 偏好对数据 | ⚠️ 不是对错，是风格 |
| **推理增强** | 可验证奖励自学推理（数学题、代码运行） | 可验证题目 | ⚠️ 无标准答案，自己试 |

### 11.7 模型包 = 权重 + config + tokenizer

| 文件 | 内容 | 占比 |
|------|------|------|
| **权重文件** | 训练产出的所有浮点数（WQ/WK/WV/WO/W_gate/W_up/W_down/lm_head……） | 99% |
| **config** | 架构说明书：多少层、每层多大、Attention 类型 | 极小 |
| **tokenizer** | 字↔编号对照表，训练时绑定 | 几 MB |

| 模型 | 权重大小 | 说明 |
|---|---|---|
| ResNet-18（图像分类） | ~45 MB | ~1100 万参数 |
| Llama-3-8B | ~16 GB（FP16） | 80 亿参数 |
| DeepSeek-V4-Pro | ~3.2 TB（FP16 等效） | 1.6T 参数，含 MoE |

**关键结论：** 文件 100 GB ≠ 训练了 100 GB 语料。文件大是因为参数个数多，不是因为塞进了训练数据。你部署的模型文件里**只有权重、config、tokenizer**，没有原始语料。

## 附录 A：Forward 伪代码（串起全流程）

```python
def forward(input_ids, past_kv_cache, weights):
    """
    input_ids:      Prefill [N]（如 [8247,15302,901,43891] 对应「我喜欢吃红烧肉」）
                    Decode  [1]（如 [21056] 对应上一步产出的 token「真」）
    past_kv_cache:  Prefill = {}
                    Decode  = 已有每层历史 K/V
    weights:        所有层的 WQ,WK,WV,WO,W_gate,W_up,W_down,lm_head，常驻 GPU 显存
    """

    hidden = embedding[input_ids]          # Prefill: [N×d]  Decode: [1×d]

    for layer_idx in range(num_layers):    # 外层循环：层数

        # ----- Attention 块 -----
        Q = hidden × WQ[layer_idx]         # [N×d] × [d×d] → [N×d]
        K = hidden × WK[layer_idx]         # [N×d] × [d×d] → [N×d]
        V = hidden × WV[layer_idx]         # [N×d] × [d×d] → [N×d]

        # 读历史 Cache → 追加新 K/V → 存回
        K_full = concat(past_kv_cache[layer_idx].K, K)   # [(hist+N)×d]
        V_full = concat(past_kv_cache[layer_idx].V, V)
        past_kv_cache[layer_idx] = (K_full, V_full)

        scores = Q × K_fullᵀ / sqrt(d)    # [N×d] × [d×(hist+N)] → [N×(hist+N)]
                                           # Prefill N tokens: [N×N]
                                           # Decode 1 token:   [1×(hist+N)]
        scores = apply_causal_mask(scores)  # Prefill 需要，Decode 跳过
        attn_weights = softmax(scores)      # 形状不变，每行归一化
        context = attn_weights × V_full    # [N×(hist+N)] × [(hist+N)×d] → [N×d]

        attn_out = context × WO[layer_idx] # [N×d] × [d×d] → [N×d]，输出投影
        hidden = attn_out + hidden         # 残差连接，形状不变 [N×d]

        # ----- FFN 块（SwiGLU）-----
        gate = silu(hidden × W_gate[layer_idx])   # [N×d] × [d×intermediate] → [N×intermediate]
        up   = hidden × W_up[layer_idx]           # [N×d] × [d×intermediate] → [N×intermediate]
        mid  = gate * up                           # 逐元素乘，形状不变 [N×intermediate]
        out  = mid × W_down[layer_idx]            # [N×intermediate] × [intermediate×d] → [N×d]
        hidden = out + hidden                      # 残差：加的是进入 FFN 前（Attention 块之后）的 hidden

    logits = hidden[-1] × lm_head         # [d] → [vocab_size]
    return logits, past_kv_cache


# --- 使用 ---
# Prefill:
logits, cache = forward(input_ids=[8247,15302,901,43891], past_kv_cache={})
next_token = sample(logits)   # → 第 1 个输出 token

# Decode:
logits, cache = forward(input_ids=[next_token], past_kv_cache=cache)
next_token = sample(logits)   # → 第 2 个输出 token
# ...重复直到 <eos>
```

**7.** 为什么需要非线性（silu + 门控乘）？去掉后多层网络会怎样？

**8.** FFN 和 Attention 各自干什么？MoE 替换的是哪块？

**9.** 权重是什么？它和训练数据是一回事吗？

**10.** 训练有哪几个阶段？每个阶段是「错了往对调」吗？

**11.** GPU 显存里主要占什么？为什么不用 CPU？

**12.** 一个 AI 产品从你发请求到看到回复，经过了哪些环节？哪些算 AI Infra？

**参考答案**

**Q1：** 输入全部已知，所有 token 堆成矩阵一次乘完（并行）。输出第 N 个字依赖第 N-1 个字的结果，不知道上一个就算不出下一个（自回归），必须串行。

**Q2：** ① Q,K,V = X × WQ/WK/WV → ② scores = Q×Kᵀ ÷ sqrt(d) → ③ 因果 mask（右上角 -∞） → ④ softmax（每行归一化） → ⑤ context = weights×V → ⑥ attn_out = context×WO。再加残差连接 attn_out+X。Q 是"我需要什么"，K 是"我能提供什么"，V 是"我的实际内容"，WO 是"把结果变换回主干网络的语言"。

**Q3：** WO 把 context（V 子空间的加权混合）变换回网络主干期望的维度方向，让后续 FFN 和下一层 Attention 能正确读取。去掉 WO，context 的方向可能跟主干网络不对齐，信息无法有效传递。

**Q4：** 不是。Prefill 跑完整 forward——每层先 Attention（顺手存 K/V）再过 FFN，缺一不可。只是 Attention 阶段顺手存了 K/V 供后续 Decode 复用。

**Q5：** 追加（append），不是重写。每步每层追加 1 行新 K/V。Decode 叫 memory-bound 因为每一步要读全部 KV Cache（可能几十 GB），但只算 1 个 token 的矩阵乘，大部分时间花在显存搬运数据上。

**Q6：** 膨胀靠两条并行路径：hidden[N×5] × W_gate[5×10] → gate[N×10]（再过 silu），hidden[N×5] × W_up[5×10] → up[N×10]；gate ⊙ up → mid[N×10]。压缩：mid[N×10] × W_down[10×5] → out[N×5]。Llama 系没有 FFN 偏置；老 GPT 用单路径 W₁→GELU→W₂，且有 b₁/b₂。

**Q7：** 没有 silu 和逐元素门控乘，多层矩阵乘数学上等价于一层（线性变换可合并），再多层都没意义。非线性让网络能表达条件逻辑（如果…则…），才能学习复杂模式。

**Q8：** Attention 让 token 之间互相通信（谁跟我有关，借多少 V），FFN 让每个 token 独立做密集计算（加工 Attention 收集来的信息，调用权重里存储的知识）。MoE 替换的是 FFN——把 FFN 拆成多个专家，路由选择激活。

**Q9：** 权重是训练产出的浮点数（如 0.0312、-1.847），几亿到几万亿个。不是训练数据——训练数据是喂给训练程序的原始文本，不进模型文件。文件大是因为参数多，不是因为塞进了语料。

**Q10：** 预训练（猜下一个字→反向传播微调，✅错了往对调）、SFT（模仿人工标注问答对，✅有标准答案）、偏好对齐（人类偏好排序，⚠️不是对错是风格）、推理增强（可验证奖励自学推理，⚠️无标准答案自己试）。

**Q11：** 权重（常驻，最大）和 KV Cache（随对话增长）。GPU 几千核心并行矩阵运算，带宽 TB/s 级；CPU 几十核心，经 PCIe 搬运太慢。权重必须常驻 GPU 显存。

**Q12：** 客户端拼 Prompt（不是）→ 鉴权/限流（不是）→ 分词（是）→ Embedding → Prefill（是）→ Decode 循环（是）→ 采样（是）→ 流式返回（是）→ Detokenize。跟模型计算直接相关的算 AI Infra。
