# AI 全链路：从打字到蹦字，每一步在算什么

## 写在前面

这份文档写给**只用过 AI 工具、但不知道背后发生了什么**的普通开发者。

你打开 Cursor 问「这段代码什么意思」，看到 AI 一个字一个字回复你——这就是你的全部认知。这份文档从这一点出发，**顺着一次请求的流动方向**，把路径上出现的每个概念当场讲清楚。不先背术语再碰计算——概念从计算过程中自然长出来。

读完你会知道：

1. 你打的那行字，经历了什么变成了 AI 的回复
2. 每个环节在算什么、为什么这么算
3. 什么是 AI Infra，为什么这个工种存在

## 1. 你打字 → 请求发出

你在 Cursor 里选中一段代码，输入「这段代码什么意思」，按下回车。

客户端把「系统提示词 + 历史对话 + 你的问题 + 代码片段」拼成 JSON，走 HTTPS 发到后端。

**到这为止，跟 AI 还没关系。** 这是通用后端——网关验身份、查额度、限流。路由层决定用哪个模型、哪台 GPU 机器。

请求真正进入 AI 链路，是从**分词**开始。

## 2. 分词：文本变数字

大模型不读汉字，只认数字。分词器（Tokenizer）把文本切成 **token**，然后映射成编号。

```
"这段代码什么意思" → [这段, 代码, 什么, 意思] → [1234, 5678, 9012, 3456]
```

> **token 是什么？** 模型处理文本的最小单位。中文里可能是一个字（「我」）、一个词（「我们」）、一个标点（「。」）。token 不一定等于完整的一个字。收费按 token 数算，速度按 tokens/s 算，显存占用跟 token 序列长度直接相关。你看到的「蹦字」，底层是「蹦 token」。

分词器跟模型是绑定的——训练时用什么分词器，推理时就必须用同一个，不能混用。GPT 和 Claude 切同一句话，token 数可能差 20%，直接影响计费和速度。

分词发生在**服务端**（API 提供方的机房），不是在你的电脑上。你发出去的是文本，到机房后才被切成编号。

**现在，你的问题变成了一串数字，比如 `[1234, 5678, 9012, 3456, 7890]`。然后呢？**

## 3. 向量化 + 模型是什么

### 3.1 数字变向量（Embedding）

每个 token 编号被映射成一个高维向量——一串浮点数。

```
1234 ("这")   → [0.12, -0.88,  1.37,  0.05, ...]
5678 ("段")   → [0.10, -0.81,  1.29,  0.03, ...]
9012 ("代码") → [-0.31, 0.45,  0.92, -1.21, ...]
...
```

语义接近的词，向量通常也更接近——"猫"和"狗"的向量距离比"猫"和"电脑"近。这个映射表是训练时学出来的。

Embedding 映射完后，模型还给每个 token 的向量附上一个**位置编码**——一小串标记「你是第几个位置」的数字，直接加到向量里。这样向量既携带了「这个词是什么意思」（语义），也携带了「这个词在句子的第几位」（位置）。位置编码对 Attention 至关重要，具体怎么起作用见第 4.3 节。

向量的长度（上面省略号里的数字个数）记作 **d**，称为**维度**。d 是模型架构设计时定下来的，写在模型包的 `config.json` 里（字段通常是 `hidden_size`），不同模型取值不同——Llama-3-8B 用 4096，Llama-3-70B 用 8192，DeepSeek V4 用 4096。d 越大，每层能编码的信息越多，但计算量也越大。

4 个 token、每个 d 维，自然形成 4 行——把它们按 token 顺序从上到下摞在一起，就是一个 **[4×d] 的矩阵**，记作 X：

```
         dim 1  dim 2  dim 3  ...  dim d
"这段"  [0.12, -0.88,  1.37,  ...,  0.05]   ← 第 1 行
"代码"  [0.10, -0.81,  1.29,  ...,  0.03]   ← 第 2 行
"什么"  [-0.31, 0.45,  0.92,  ..., -1.21]   ← 第 3 行
"意思"  [1.02,  0.13, -0.76,  ...,  0.55]   ← 第 4 行
```

后面说的 **X[4×d]**，就是这个意思——4 个 token 的向量堆成的矩阵，4 行 d 列。GPU 做矩阵乘法的时候，4 行是一起算的。这个矩阵 X 接下来就要进入模型。

### 3.2 模型是什么

模型就是一个巨大的数学函数：

```
输出 = f(输入, 权重)
```

拆开是两个东西：

| | 是什么 | 怎么来的 | 变不变 |
|---|---|---|---|
| **架构 f**（神经网络） | 运算流水线——多少层、每层做什么类型的运算 | 人设计的 | 固定 |
| **权重** | 流水线上每个操作的具体数字——乘几、加几 | 训练出来的 | 同一架构不同训练产出不同权重 |

**类比：** 菜谱步骤（先切再炒最后调味）= 架构。盐 3g、糖 5g = 权重。同一份菜谱，不同厨师练出来的用量不同，味道就不一样。

**神经网络**就是那个「运算流水线」——数据从一头进去，经过几十到上百「层」，每层做一批固定类型的数学运算（主要是乘法和加法），从另一头出来。为什么叫"神经"？1940 年代的研究者从大脑神经元「接收信号→超过阈值→放电」获得灵感，但今天的深度神经网络跟生物神经元基本没关系——就是一个极深的、可训练的数学函数。

### 3.3 权重怎么来的（轻量版）

权重是**训练**出来的。训练程序做的事极其简单：**给定前面的字，猜下一个字是什么。**

```
输入："今天天气"  → 模型猜 "很"  → 对比正确答案 → 猜错了
       ↓
反向传播：从最后一层往回，精确算出每个权重对猜错贡献了多少
       ↓
微调所有权重，往减小误差的方向拧一点点
       ↓
重复万亿次，直到模型猜下一个字很准
```

这就是**预训练（Pre-training）**——产出的是 Base Model，会接话但不会对话。后面还要经过 SFT（学对话格式）、偏好对齐（学风格）、推理增强（学思考）。详细训练流程见第 11 节。

产出的**模型包**包含三个文件：

| 文件 | 内容 | 占比 |
|------|------|------|
| **权重文件** | 所有浮点数 | 99% |
| **config** | 架构说明书（多少层、每层多大） | 极小 |
| **tokenizer** | 字↔编号对照表 | 几 MB |

**你的 5 个 token 的向量矩阵，现在就进入这个模型。第一件事是 Attention。**

## 4. Attention：token 之间互相"看"

### 4.1 Attention 是什么

矩阵 X[4×d] 进入模型，面对一个问题：每个 token 的向量目前只包含自己词典里的意思——「代码」只知道自己是"代码"，不知道前面有「这段」在限定它，也不知道后面有「什么」在问它。

模型需要让每个 token **看到并加权利用上下文中的其他 token**。这个机制就是 Attention。

具体做法：拿 X 分别乘三组权重矩阵 Wq、Wk、Wv，把同一个 X 投影到三个不同的语义空间，得到 Q、K、V：

```
Q = X × Wq    投影到 Query「需求空间」：这个向量"想找什么"
K = X × Wk    投影到 Key  「特征空间」：这个向量"是什么、有什么属性"
V = X × Wv    投影到 Value「内容空间」：这个向量"携带的实际信息"
```

为什么需要 Q 和 K 分开？因为「我想找什么」和「我是什么」是两种信息角度。同一个词「红烧肉」——作为 Q 时表达「需要找到跟吃相关的东西」，作为 K 时表达「一道菜、甜的、五花肉做的」。Q 和 K 做点积，本质上是在问：**需求空间里的"想要什么"和特征空间里的"能提供什么"对上了吗？** 对上了，分数就高，那个位置的 V 就被拿得多。

**Wq、Wk、Wv 就是第 3.2 节说的「权重」——训练时学出来的浮点数矩阵，存在模型包的权重文件里，推理服务启动时 load 进 GPU 显存。** 每层 Attention 有自己的一组，50 层就有 50 组。它们和 Embedding 映射表、以及后面第 5 节要讲的 FFN 的权重加在一起，就是对外的「模型大小」——Llama-3-8B 那 16GB 就是这些矩阵的总和。

用「这段代码什么意思」4 个 token 来看 Q×Kᵀ 的结果（数字是示意）：

```
              这段   代码   什么   意思
Q("这段")    [0.9   0.3   0.2   0.1 ]  ← 每一格都算出来了
Q("代码")    [0.3   0.8   0.4   0.1 ]
Q("什么")    [0.2   0.4   0.7   0.5 ]
Q("意思")    [0.1   0.2   0.5   0.8 ]
```

注意右上角——"这段"的 Q 跟"代码""什么""意思"的 K 实际上**已经算出了分数**。但语言模型的任务是「根据前面的字猜下一个」：如果训练时"这段"能看到后面的"代码"，模型直接抄答案就行，不需要猜。所以模型在 softmax 之前**故意**把右上角（未来位置）的分数改成 `-∞`：

```
              这段   代码   什么   意思
Q("这段")    [0.9   -∞    -∞    -∞ ]  ← 主动遮住
Q("代码")    [0.3   0.8   -∞    -∞ ]
Q("什么")    [0.2   0.4   0.7   -∞ ]
Q("意思")    [0.1   0.2   0.5   0.8 ]  ← 只有最后一行看全了
```

softmax 遇到 `-∞` 会变成 0——未来 token 的 V 被彻底清零。**不是数据访问不到，是权重被人为压成 0。** 这就同时做到了：四个 token 并行计算（Q×Kᵀ 整张表一次出），每个位置在权重上只用得到前面的 token。

越靠后的位置能看到越多——最后一个位置是唯一看全了整句话的，所以 Prefill 结束后取**末尾向量**做预测。

### 4.2 完整计算过程：五步走一遍

用简单例子「我喜欢吃红烧肉」4 个 token，每个 d=16 维，堆成 X[4×16]。

**第一步：一次矩阵乘法同时算出 Q、K、V。**

```
Q = X × Wq → [4×16]
K = X × Wk → [4×16]
V = X × Wv → [4×16]
```

没有 for 循环。4 个 token 堆成矩阵，一次矩阵乘全出来。GPU 几千个核心同时干这件事。

**第二步：算相关度分数。**

```
scores = Q × Kᵀ → [4×4]

每格 = tokenᵢ 的 Q · tokenⱼ 的 K

              K₁("我")  K₂("喜欢") K₃("吃")  K₄("红烧肉")
Q₁("我")      [ 0.9      0.3        0.1       0.0    ]
Q₂("喜欢")    [ 0.3      0.8        0.4       0.1    ]
Q₃("吃")      [ 0.2      0.4        0.7       0.5    ]
Q₄("红烧肉")  [ 0.1      0.2        0.5       0.8    ]
```

**第三步：因果 mask——后面的不能偷看。**

右上角填 `-∞`：

```
scores + mask:

              K₁    K₂    K₃    K₄
Q₁("我")      [0.9  -∞    -∞    -∞  ]  ← 只能看自己
Q₂("喜欢")    [0.3  0.8   -∞    -∞  ]  ← 看"我"和自己
Q₃("吃")      [0.2  0.4   0.7   -∞  ]  ← 看"我""喜欢"和自己
Q₄("红烧肉")  [0.1  0.2   0.5   0.8 ]  ← 可以看全部
```

**第四步：softmax 变成权重。**

每行做 softmax，`-∞` 变 0：

```
weights:

              K₁     K₂     K₃     K₄
Q₁("我")      [1.00  0.00   0.00   0.00]
Q₂("喜欢")    [0.38  0.62   0.00   0.00]
Q₃("吃")      [0.27  0.33   0.40   0.00]
Q₄("红烧肉")  [0.20  0.22   0.27   0.31]
```

**第五步：用权重把 V 汇总。**

```
output = weights × V → [4×16]
```

"红烧肉"那一行 = 0.20×V("我") + 0.22×V("喜欢") + 0.27×V("吃") + 0.31×V("红烧肉")。"我"那一行 = 1.00×V("我")（看不到后面）。

这个 output 就是 Attention 的最终产出——它会继续往下走，过 FFN、传到下一层。

以上五步就是 Attention 的一次完整计算。与此同时，第一步算出的 K 和 V 还做了一件事——顺手存进 KV Cache：

```
KV Cache[当前层] = { K: [4×16], V: [4×16] }
```

**KV Cache 只存 K 和 V，不存 output。** 存 K/V 纯粹是为了 Decode——后续每生成一个新 token，历史 token 的 K 和 V 不用重算，直接从 Cache 读出来拼接。而 output 会继续往前走——过 FFN、传到下一层。FFN 在第 5 节马上讲。

### 4.3 总结

Attention 不是「一个动作」，是**一套计算块**：QKV 投影 → 分数矩阵 → mask → softmax → 加权 V。所有 token 的 Q、K、V 是矩阵乘法同时算的，没有逐个 token 的循环。

#### 补充：位置和语序的影响

前面第 3.1 节提到，Embedding 之后给每个向量附上了**位置编码**。这个位置信息正是在 Attention 这一步发挥作用的——位置编码已经混在 X 里了，所以 Q 和 K 都带着位置信号。Q×Kᵀ 算出来的分数，既反映了「语义上有多相关」，也反映了「位置上是什么关系」。

因果 mask 决定了信息只能从左往右流——前面的 token 看不到后面。这意味着语序直接影响「当前 token 能看到哪些上下文」：

```
「我吃了红烧肉」→ 到「红烧肉」时，已经知道主语"我"和动作"吃"
「红烧肉被我吃了」→ 到「红烧肉」时，还不知道后面会发生什么
```

同一个词在不同语序下，Attention 融合的上文不同，输出向量自然不同。不同语言（SVO 中文、SOV 日语）的词序差异，模型是在训练中通过海量语料的统计分布学会的——它不记语法规则，但统计上知道每种语序模式下哪个位置该关注什么。

## 5. 一层不够：Attention + FFN = 一个 Transformer 层

### 5.1 FFN 是什么

拿 4.2 节第五步算出来的结果来说——`output = weights × V → [4×d]`。以「红烧肉」这个 token 为例，它经过 Attention 后的向量是：

```
"红烧肉"Attention后 = 0.20×V("我") + 0.22×V("喜欢") + 0.27×V("吃") + 0.31×V("红烧肉")
                      ↑ 这是一个 d 维向量，融合了整句上下文。但注意——这些信息只是被简单加权混合了
```

**FFN 的目的：加工混合后的信息。** Attention 让 token 互相通信，把别人的 V 拉了一部分过来。但这些信息是被生硬地加权平均在一起的，还没有被深入理解和转化。FFN 做的事：拿这个混合向量，通过两层全连接把里面的信息"压榨"一遍——膨胀到更大的维度去充分展开、用激活函数做非线性筛选、再压缩回来。经过这一道，向量里的信息不再是简单的加权混合，而是被模型用自己的"知识"重新编码过。

> 实际上，模型中大量的事实性知识主要存储在 FFN 的权重里——"巴黎是法国的首都""`import numpy as np`"这类信息，Attention 负责在 token 之间传递，FFN 负责在自己的权重里"回忆"和"运用"。

d=16，假设这一层 FFN 的中间维度设为 48（3 倍膨胀；实际模型膨胀倍数不同，原理一样）。下面不走矩阵符号，每一步都说明**进去的是什么形状、出来的是什么形状**。

**起点：**「红烧肉」经过 Attention 后的向量 = 长度为 16 的数组。

```
[0.2, -0.5, 1.3, ..., 0.7]   ← 16 个数字
```

**第一步：数组[16] × W₁[16×48] → 数组[48]。**

W₁ 是一个 16 行、48 列的表格。数组乘表格 = 数组的 16 个数字分别去"扫描"表格的每一列，每列扫完得出 1 个数字，48 列 = 48 个数字：

```
输出[1] = 0.2×w₁₁ + (-0.5)×w₂₁ + 1.3×w₃₁ + ... + 0.7×w₁₆₁
输出[2] = 0.2×w₁₂ + (-0.5)×w₂₂ + 1.3×w₃₂ + ... + 0.7×w₁₆₂
...（重复 48 次）
输出[48] = ...
```

出来的是一个长度为 48 的数组。维度膨胀就发生在这里——16 个数字变成了 48 个。

**第二步：数组[48] + b₁[48] → 数组[48]。**

b₁ 也是一个长度为 48 的数组。两个等长数组相加，就是每个位置对位相加：`结果[i] = 上一步输出[i] + b₁[i]`。形状不变。

**第三步：过激活函数 SwiGLU → 数组[48]。**

48 个数字，每个独立过激活函数。形状不变。

**第四步：数组[48] × W₂[48×16] → 数组[16]。**

跟第一步对称——W₂ 是 48 行、16 列的表格，每列扫完得 1 个数字，16 列 = 16 个数字。48 个数字压缩回 16 个。

**第五步：数组[16] + b₂[16] → 数组[16]。**

b₂ 是长度为 16 的数组，对位相加。形状不变。

**全程形状变化：16 → 48 → 48 → 48 → 16 → 16。进去一个长度为 16 的数组，出来还是一个长度为 16 的数组。**

---

**W₁/W₂/b₁/b₂ 的形状由什么决定？** 两个架构参数：d 和 `intermediate_dim`，都写在 config.json 里。

```
W₁: d 行 × intermediate_dim 列  
b₁: 长度 intermediate_dim       
W₂: intermediate_dim 行 × d 列  
b₂: 长度 d                      
```

形状训练前确定、训练后固定。训练学的是表格**里面的数字**。

**W₁/W₂ 的形状跟输入 token 数量毫无关系。** 不管来 4 个 token 还是 4000 个，每个 token 的向量都乘同一个 W₁[16×48]。前面 `X[4×d]` 的 4 是 token 数量（每次请求可变），W₁ 的 16×48 是架构写死的（不变）。

---

**"全连接"的"全"在哪？** 回头看第一步 `数组[16] × W₁[16×48] → 数组[48]`。输出的 48 个数字，每个都是从**全部** 16 个输入加权求和算出来的——`输出[1] = 输入[1]×w₁₁ + 输入[2]×w₂₁ + ... + 输入[16]×w₁₆₁`。没有哪个输入被跳过，没有哪个权重被两个输出共享。16×48=768 个权重，各算各的。实际模型中 d=4096、中间=14336，这一步就有 ~5900 万个独立权重。

对比下一节要讲的 MoE——MoE 把这一层的权重拆成多个专家，每个 token 只激活其中几个。标准 FFN 是「全」（全部输入→全部输出），MoE 是「稀疏」（全部输入→部分专家→全部输出）。

**所有神经网络都是这样算的吗？** 骨架一样：线性变换（乘表格 + 加偏置）→ 非线性激活。这是深度学习的通用配方——没有激活函数，多层矩阵乘会退化成单层，深度就失去了意义。不同架构区别在于表格的形状和连接模式，但乘表格 + 激活这个配方不变。

**ReLU 和 SwiGLU 的区别？** ReLU：正数原样通过，负数直接变 0。SwiGLU：把输入拆成两路——一路算"值"，一路算"门控"（决定放多少过去），两路逐元素相乘。SwiGLU 允许少量负值通过，信息保留更丰富。Llama、DeepSeek、Qwen 几乎都用 SwiGLU 或其变体。

```
ReLU:   每个数字独立做 max(0, x)            — 负数直接扔掉
SwiGLU: 每个数字 = (xW_val) × sigmoid(xW_gate)  — 有一道门控制通过量
```

### 5.2 MoE 替换的是 FFN

标准 FFN 的 W₁ 和 W₂ 对所有 token 一视同仁——「红烧肉」乘的 W₁ 和「我」乘的 W₁ 是同一个矩阵。MoE（Mixture of Experts，混合专家）的思路：不做一份巨大的 W₁/W₂，而是准备多份较小的"专家"，让路由器根据 token 的内容决定分配给哪几个专家。

```
标准 FFN：                      MoE FFN：
每个 token 过同一组 W₁,W₂       每个 token 先经过路由器
                               ↓
                              路由器看 token 内容："红烧肉？分给专家3、7、15"
                               ↓
                              只有被选中的专家做计算，其余不动
                               ↓
                              各专家输出加权合并
```

- 标准 FFN：一份 W₁（比如 [16×48]），每个 token 都用它
- MoE：比如 256 个专家，每个专家的 W₁ 是 [16×?]（缩小到约 1/256），每个 token 只激活其中 8-9 个

**关键效果：** 总参数量可以极大（256 个专家各有各的权重），但每个 token 激活的计算量只相当于一个中等大小的 FFN。DeepSeek V4 总参数 1.6T，每次只激活 49B——~3%。

**MoE 和 Attention 不冲突——** MoE 换的是 FFN 那块，Attention 照常运行。

### 5.3 多层堆叠：从字面到意图

一层 Attention 不够。实际模型堆几十到上百层（Llama-3-8B 有 32 层），每层在前一层的基础上继续加工。不同深度的层编码了不同层次的信息——这是研究者用**探针实验**（在每层输出的向量上训练分类器，看能识别出什么信息）验证过的：

> 「小明说他昨天去了那家新开的川菜馆，菜很辣但是他很喜欢」
>
> 追踪最后一个 token「喜欢」在各层编码了什么：

```
第 1-4 层（表层编码）：
  向量里能检测到的："喜欢"前面紧挨着"很"，再往前是"他"
  检测不到的：这个"他"是谁、整句的结构如何

第 5-12 层（句法编码）：
  能检测到的：句子由两大块组成；"但是"标记了转折；"他"与"小明"共享指代
  往前推进了：不再只是相邻字的关系，而是识别出了句子的骨架

第 13-24 层（语义编码）：
  能检测到的："辣"和"喜欢"被"但是"隔开，情绪方向相反——构成矛盾
  往前推进了：不只是谁修饰谁，而是开始编码语义冲突

第 25-32 层（语用编码）：
  能检测到的：矛盾被消解——"喜欢"是重心，"辣"被重新编码为正面信号
  往前推进了：整体情绪是正面的，意图是推荐而非抱怨
```

> **诚实说明：** 各层输出的仍然是向量（一串数字），不是自然语言。"学到了什么"是研究者用探针实验反推出来的。深浅层的边界也不是一刀切——整个网络是连续地、渐进地丰富向量里携带的信息。

现在，每一层要做什么已经清楚了——Attention 让 token 互相通信，FFN 各自加工，一层一层往上堆。接下来看这几十层在推理时怎么跑起来：先是 Prefill，输入一次过完所有层。

---

## 6. Prefill：输入一次过完所有层

分词器把用户输入变成 token 列表，经过 Embedding 变成向量矩阵 X 后，模型进入 **Prefill 阶段**。

**一句话：输入 token 列表 → 第 1 层 Attention+FFN → 第 2 层 Attention+FFN → ... → 第 L 层 Attention+FFN → 末尾采样 → 第 1 个输出 token。** 模型有多少层，就串多少轮。每层内部是第 4、5 节讲的那一套——Attention 让 token 互相通信（顺手存 K/V 到 Cache），FFN 各自加工。KV Cache 是 Attention 阶段顺手存的副产品，不是 Prefill 的目的。

末尾采样就是：所有层走完后，取最后一个位置的向量 `output[-1]`（长度为 d 的数组，比如 16 个数字），乘一个权重矩阵 lm_head[16 × 100000]，产出一个长度为词表大小（比如 100000）的分数数组，softmax 变成概率分布，从里面挑一个作为第 1 个输出 token。lm_head 跟 Wq、W₁ 一样是训练学出来的权重，形状 [d × vocab_size] 写在 config.json 里。softmax 的原理和采样的策略（贪心、Temperature、Top-p）细节见第 8 节。

下面用一个具体例子走一遍——「我喜欢吃红烧肉」4 个 token、模型 3 层（实际模型有几十上百层，逻辑一样，3 层够看清结构）：

```
Prefill 开始
  输入: 4 个 token 的向量, X₀ = [4×16]
  KV Cache: 空
       ↓
第 1 层
  Attention: 从 X₀ 算 QKV → scores[4×4] → mask → softmax → ×V
  💾 存 KV Cache[0]: 基于 X₀ 算出的 K[4×16], V[4×16]
  FFN: 4 个 token 各自过密集变换
  输出: X₁[4×16] → 传第 2 层
       ↓
第 2 层
  Attention: 从 X₁ 算 QKV（不是重算原始 X₀，是拿上一层加工过的 X₁）
  💾 存 KV Cache[1]: 基于 X₁ 算出的 K[4×16], V[4×16]
  FFN → 输出 X₂[4×16] → 传第 3 层
       ↓
第 3 层
  从 X₂ 算 QKV → ...
  💾 存 KV Cache[2]: 基于 X₂ 算出的 K[4×16], V[4×16]
  FFN → 输出 X₃[4×16]
       ↓
末尾采样
  取最后一个位置（"红烧肉"）的向量 X₃[-1] = 长度为 16 的数组
  → 乘 lm_head[16×100000] → 长度为 100000 的分数数组（logits）
  → softmax → 100000 个概率（全正，和为 1）
  → 按策略挑一个 → 第 1 个输出 token，比如 "很"
```

**Prefill 结束。产物：** 第 1 个输出 token；KV Cache = L 层 × N 行 × (K+V)，每层的 K/V 来自该层的输入。

**关键点：** Attention 和 FFN 在计算上都是矩阵并行——所有 token 堆成矩阵，一次矩阵乘全出，没有 `for token in tokens` 这种串行循环。FFN 里「每个 token 独立」指的是信息不交叉（红烧肉的 FFN 结果不混入我的信息），不是说逐个串行算。Prefill 是完整 forward，只产出 1 个 token。

### KV Cache 到底占多大

用真实数字感受一下：

```
输入 token 数 N = 1000
层数 L = 50
每层 KV 维度 d = 1024

KV Cache = L × N × d × 2(K+V) × 2字节(FP16)
         = 50 × 1000 × 1024 × 2 × 2
         ≈ 200 MB（一个请求）
```

100 个用户同时来 ≈ 20 GB，光 KV Cache，不算权重。

## 7. Decode：一个字一个字蹦

Prefill 产出了第 1 个输出 token（"很"）和全量 KV Cache（3 层 × 4 行）。之后进入 **Decode 循环**。

**Decode 做的事：拿上一个产出的 token（只是一个编号），走一遍完整的 Embedding + L 层 Attention+FFN + 末尾采样，产出下一个 token。跟 Prefill 的关键区别：**
- **输入形状**：Prefill 全程是矩阵 [N×d] 进、[N×d] 出；Decode 全程是向量 [1×d] 进、[1×d] 出
- **K/V 来源**：Prefill 每层的 K/V 当场从输入全量算；Decode 每层只算新 token 自己的 K/V，历史部分从 Cache 读出来拼接
- **因果 mask**：Prefill 需要（N 个 token 有前后关系）；Decode 不需要（新 token 是最后一个，后面没人）

### Decode 第 1 步：从 "很" 到 "好"

Prefill 产出的 "很" 只是一个 token 编号（比如 9999）。跟第 2 节一样，先过 Embedding：

```
9999 → Embedding 查表 → 长度为 16 的数组 X_new = [0.31, -0.22, 0.87, ..., -0.15]
                          ↑ 16 个数字，跟 Prefill 的 d=16 一致
```

这个数组进入第 1 层。每层的 Attention 和 FFN 使用的权重 Wq、Wk、Wv、W₁、W₂ 跟 Prefill 是同一套，不变。

**第 1 层：**

```
输入: X_new = [1×16]（只有一个 token 的向量，不是 Prefill 的 [4×16]）

Attention:
  Q = X_new × Wq → [1×16]      ← 只有 1 行，不是 Prefill 的 4 行
  K = X_new × Wk → [1×16]      ← 同上
  V = X_new × Wv → [1×16]

  💾 KV Cache 操作:
     读 Cache[0]: Prefill 存的 K[4×16], V[4×16]
     把刚算的 K[1×16] 拼到末尾 → K_full = [5×16]
     把刚算的 V[1×16] 拼到末尾 → V_full = [5×16]
     存回 Cache[0] 供下一步用

  Attention 计算:
     scores = Q[1×16] × K_fullᵀ[16×5] → [1×5]
     ↑ 新 token 的 Q 跟全部 5 个 token 的 K 做点积
     不需要 mask——新 token 是最后一个位置，后面没人

     weights = softmax(scores) → [1×5]
     ↑ 5 个权重，分别对应 4 个历史 token + 自己

     attn_out = weights[1×5] × V_full[5×16] → [1×16]
     ↑ 加权汇总所有 token 的 V，得到新 token 融合了历史上下文的新表示

FFN:
  跟 Prefill 完全相同: 数组[16] → W₁[16×48] → 数组[48] → SwiGLU → W₂[48×16] → 数组[16]

输出: [1×16] → 传第 2 层
```

**第 2 层：**

```
输入: 上一层 FFN 的输出 [1×16]

Attention:
  Q,K,V = 输入 × Wq,Wk,Wv → 各 [1×16]

  💾 读 Cache[1]（Prefill 存的 4 行）→ 拼上刚算的 1 行 → 5 行 → 存回

  scores = Q[1×16] × K_fullᵀ[16×5] → [1×5]
  weights = softmax(scores)
  attn_out = weights × V_full[5×16] → [1×16]

FFN → 输出 [1×16] → 传第 3 层
```

**第 3 层：同第 2 层。** 读 Cache[2]，追加，用全量算 Attention，FFN，输出 [1×16]。

```
末尾采样:
  取最后一层的输出 [1×16]（只有一个 token，就是它自己）
  → × lm_head[16×100000] → [1×100000] 个 logits
  → softmax → 概率分布
  → 采样 → 第 2 个输出 token，比如 "好"
```

### Decode 第 2 步及之后

```
输入: "好"（token 编号）→ Embedding → [1×16]
每层 Cache 从 5 行 → 6 行
...同样的 Embedding + L 层 Attention+FFN + 采样
产出: 第 3 个输出 token

重复直到产出结束 token（<eos>）或达到长度上限
```

### 为什么输出必须串行

第 N 个 token 依赖第 N-1 个 token——不知道上一个字是什么就算不出下一个。这叫**自回归**（Autoregressive），本质是超大号 Autocomplete：每次根据已有内容猜下一个最可能的字。

### Prefill vs Decode 对比

| | Prefill | Decode（每一步） |
|---|---|---|
| **输入大小** | N 个 token | 1 个 token |
| **每层 Attention 输入** | Q[N×d], K[N×d], V[N×d] | Q[1×d], K 从 Cache 读 [(N+k)×d] |
| **KV Cache** | 新建，每层写 N 行 | 追加，每层加 1 行 |
| **因果 mask** | 需要（N 个 token 有前后关系） | 不需要（新 token 是最后一个） |
| **并行度** | 高——N 个 token 同时算 | 低——只有 1 个 token |
| **瓶颈** | 计算密集型：时间花在矩阵乘法上 | 内存密集型：每步读全部 KV Cache（几十 GB），只算 1 个 token，时间花在搬运数据上 |
| **产出** | 第 1 个输出 token | 每步 1 个输出 token |

## 8. 采样：分数变文字

无论 Prefill 还是 Decode，所有层走完后，拿到最后一个位置的向量 `output[-1]`——长度为 d 的数组。第 6 节 Prefill 末尾，X₃[-1] 是"红烧肉"位置的 16 维向量；第 7 节 Decode 每步末尾，同样取最后一个位置的向量。接下来把它变成一个具体的 token。

### 8.1 lm_head：从 d 维向量到词表分数

**lm_head 是什么？** 一个训练学出来的权重矩阵，存在模型包的权重文件里，推理时 load 进 GPU。跟 Wq、W₁ 一样是权重，专门负责最后这一步。

**形状：** `[d × vocab_size]`。d 行（跟模型维度一致），vocab_size 列（每列对应词表里的一个候选 token）。d 和 vocab_size 都写在 config.json 里，架构写死的。

```
最后一个位置的向量 [1×d] × lm_head [d×vocab_size] = [1×vocab_size]

比如 d=16, vocab_size=100000:
  [1×16] × [16×100000] = [1×100000]
  出来 100000 个分数，每个对应词表里一个 token
```

**词表跟 lm_head 不是一个东西。** 词表是 tokenizer 里的查表规则——编号 1234 对应汉字"这"，存的是映射关系。lm_head 是权重矩阵，存的是训练学出来的浮点数。两个都在模型包里，各是各的。

### 8.2 softmax：把分数变成概率

lm_head 产出的 100000 个分数（叫 logits），可以是任意实数——正的负的大的小的。需要变成概率才能采样。softmax 做的事：

```
输入: [3.2,  1.5,  -0.8,  0.1]     ← 任意实数，有正有负

第1步: 每个数做 eˣ（指数函数）
       e^3.2 ≈ 24.5                 ← 正数变得更大
       e^1.5 ≈ 4.5
       e^-0.8 ≈ 0.45                ← 负数不会消失，只是变小
       e^0.1 ≈ 1.1

第2步: 求和  24.5 + 4.5 + 0.45 + 1.1 = 30.55

第3步: 每个除以总和
       24.5 / 30.55 ≈ 0.80
       4.5 / 30.55 ≈ 0.15
       0.45 / 30.55 ≈ 0.01
       1.1 / 30.55 ≈ 0.04

输出: [0.80, 0.15, 0.01, 0.04]     ← 全正、和为 1，就是概率分布
```

> 回头看 Attention 第四步——那里也是 softmax，但输入里有 `-∞`（mask 加的）。`e^(-∞) = 0`，所以未来位置的概率变成 0。正常 softmax 没有 `-∞` 输入，负数会变成小的正概率，不会消失。

### 8.3 从概率分布里挑一个

100000 个候选，每个有一个概率。怎么挑：

- **贪心（Greedy）：** 永远选概率最高的——稳定但死板，容易重复
- **Temperature：** 采样前先把分数除以温度值。温度 > 1 → 概率更均匀 → 更随机（更有创意）。温度 < 1 → 概率更集中 → 更保守
- **Top-p：** 按概率从高到低累加，只在前 p% 的候选中随机挑——折中方案

挑出来的结果是一个 token 编号，通过**分词器查表**还原成可读文字（Detokenize）。

## 9. 流式返回：你看到字蹦出来

Decode 每产出一个 token，就通过 SSE（Server-Sent Events，HTTP 长连接）推送给客户端。客户端收到一段渲染一段——所以你看到「一个字一个字蹦出来」。

## 10. 全景回顾

### 完整链路

```text
你打字 → 客户端拼 JSON → HTTPS → 网关(鉴权/限流)
  → 路由(选模型/机器) → 分词(文本→token编号)
  → Prefill(输入一次过所有层,建KV Cache,产第1个token)
  → Decode循环(每次1 token, Cache追加,采样) → 流式推回
  → Detokenize(token编号→文字) → 你看到字蹦出来
```

### AI Infra 边界

```
❌ 不是 AI Infra: 客户端UI / 拼Prompt / 鉴权限流计费
✅ 是 AI Infra:   分词 → 推理引擎+框架 → KV Cache管理 → GPU调度 → 采样 → 流式推送
```

### GPU 显存里有什么

| 占用项 | 大小趋势 | 说明 |
|---|---|---|
| **权重** | 固定，最大 | 7B 模型约 14GB(FP16)，70B 约 140GB |
| **KV Cache** | 随对话增长 | 每多 1 token，各层多存一行 K/V；长对话 + 高并发 = 显存杀手 |
| **临时缓冲区** | 单次 forward | 算完释放 |
| **token 编号** | 极小 | 可忽略 |

### 为什么用 GPU 不用 CPU

GPU 有几千个计算核心，擅长同时做大量矩阵乘法——这正是 Attention 和 FFN 需要的。CPU 只有几十个核心。更关键的是：GPU 显存带宽极高（TB/s 级），权重必须常驻显存——如果每次 forward 从 CPU 内存通过 PCIe 总线搬运到 GPU，整体会慢几个数量级。

> **FP16 / 量化：** FP32 每个数占 4 字节，FP16 占 2 字节，INT8 占 1 字节。把 FP32 权重转成低精度存储叫**量化**——能省一半甚至更多显存，代价是轻微精度损失。

## 11. 训练：权重到底怎么练出来的

前面第 3.3 节给了轻量版。这里展开——一个大模型从零到能对话，经过四个阶段。

### 11.1 预训练（Pre-training）

**任务：** 海量文本上猜下一个字。

```
随机抽一段文本 → 给前面 N 个字让模型猜第 N+1 个字
  → 对比正确答案 → 猜错了
  → 反向传播：从最后一层往回，精确算出每个权重对猜错贡献了多少
  → 微调所有权重（贡献大的多拧，贡献小的少拧）
  → 重复万亿次
```

**数据：** 《三体》全文、维基百科、Stack Overflow、GitHub 公开代码、新闻网页……数万亿 token。这些文本**不进模型文件**——训练跑完后，模型文件里只有训练产出的数字（权重）。训练数据留在算法团队的集群里。

**产出：** Base Model——会接话，但不会对话。你问「这段代码什么意思」，它可能继续往下写代码而不是解释，因为它只学会了「猜下一个字」，还没学会「回答问题的格式」。

**成本：** 几万亿 token、成百上千张 GPU、几周到几个月、几百万到几千万美元。

### 11.2 SFT（监督微调，Supervised Fine-Tuning）

用人工写的「问题→标准答案」对子（通常 1 千到 5 万条）继续训练，让模型学会对话格式和遵循指令。

```
输入: "请解释这段代码的功能"
目标输出: "这段代码实现了一个二分查找算法，时间复杂度 O(logn)..."
```

这一步后模型学会了「用户问问题→我回答」的格式。但风格可能还很生硬。

### 11.3 偏好对齐（RLHF / DPO）

同一个问题给多个回答，人类标注哪个更好。模型学习「为什么 A 优于 B」——不再是简单对错判断。DPO（Direct Preference Optimization，直接偏好优化）是 2024-2025 年的主流做法：跳过训练独立奖励模型的步骤，直接从偏好数据里学，计算量减半。

### 11.4 推理增强（GRPO / RLVR）

DeepSeek-R1 用的方法。用数学题答案、代码运行结果等**可验证的奖励信号**，让模型在试错中自己学会推理——没人教它「怎么思考」，它在强化学习过程中自己涌现出「先分析再回答」的能力，甚至出现「aha moment」——模型在中间步骤自己停下来、发现前面错了、修正后再继续。

### 11.5 训练阶段一览

| 阶段 | 做什么 | 数据规模 | 是「错了往对调」吗 |
|------|--------|----------|:---:|
| **预训练** | 猜下一个字，猜错→反向传播微调 | 数万亿 token | ✅ |
| **SFT** | 人工标注问答对，模仿标准答案 | 1K~50K 条 | ✅ |
| **偏好对齐** | 人类偏好排序，学"哪个答案更好" | 偏好对数据 | ⚠️ 不是对错，是风格 |
| **推理增强** | 可验证奖励自学推理（数学题、代码运行） | 可验证题目 | ⚠️ 无标准答案，自己试 |

**架构趋势（2025-2026）：**

### 11.7 模型包 = 权重 + config + tokenizer

| 文件 | 内容 | 占比 |
|------|------|------|
| **权重文件** | 训练产出的所有浮点数 | 99% |
| **config** | 架构说明书：多少层、每层多大、Attention 类型 | 极小 |
| **tokenizer** | 字↔编号对照表，训练时绑定 | 几 MB |

| 模型 | 权重大小 | 说明 |
|---|---|---|
| ResNet-18（图像分类） | ~45 MB | ~1100 万参数 |
| Llama-3-8B | ~16 GB（FP16） | 80 亿参数 |
| DeepSeek-V4-Pro | ~3.2 TB（FP16 等效） | 1.6T 参数，含 MoE |

**关键结论：** 文件 100 GB ≠ 训练了 100 GB 语料。文件大是因为参数个数多，不是因为塞进了训练数据。你部署的模型文件里**只有权重、config、tokenizer**，没有原始语料。

## 附录 A：Forward 伪代码（串起全流程）

```python
def forward(input_ids, past_kv_cache, weights):
    """
    input_ids:      Prefill [N]（如 [1234,5678,9012,3456]）
                    Decode  [1]（如 [9999]）
    past_kv_cache:  Prefill = {}
                    Decode  = 已有每层历史 K/V
    weights:        所有层的 Wq,Wk,Wv,FFN,lm_head，常驻 GPU 显存
    """

    hidden = embedding[input_ids]          # Prefill: [N×d]  Decode: [1×d]

    for layer_idx in range(num_layers):    # 外层循环：层数

        # ----- Attention 块 -----
        Q = hidden × Wq[layer_idx]         # 矩阵乘，同时算全部 token
        K = hidden × Wk[layer_idx]
        V = hidden × Wv[layer_idx]

        # 读历史 Cache → 追加新 K/V → 存回
        K_full = concat(past_kv_cache[layer_idx].K, K)
        V_full = concat(past_kv_cache[layer_idx].V, V)
        past_kv_cache[layer_idx] = (K_full, V_full)

        scores = Q × K_fullᵀ / sqrt(d_head)  # Prefill: [N×N]  Decode: [1×(N+k)]
        scores = apply_causal_mask(scores)     # 右上角 = -∞
        weights = softmax(scores)
        attn_out = weights × V_full

        hidden = attn_out + hidden             # 残差连接

        # ----- FFN 块 -----
        hidden = FFN(hidden)                   # MoE 替换的就是这里
        hidden = hidden + attn_out

    logits = hidden[-1] × lm_head             # [d] → [vocab_size]
    return logits, past_kv_cache


# --- 使用 ---
# Prefill:
logits, cache = forward(input_ids=[1234,5678,9012,3456], past_kv_cache={})
next_token = sample(logits)   # → 第 1 个输出 token

# Decode:
logits, cache = forward(input_ids=[next_token], past_kv_cache=cache)
next_token = sample(logits)   # → 第 2 个输出 token
# ...重复直到 <eos>
```

## 附录 D：自检清单

**2.** Attention 的 5 步计算过程分别是什么？Q、K、V 是什么关系？

**3.** Prefill 阶段是不是只算 KV Cache？它还做了什么？

**8.** GPU 显存里主要占什么？为什么不用 CPU？

**9.** 「推理部署工程师」和「AI 算法工程师」有什么区别？

**10.** 一个 AI 产品从你发请求到看到回复，经过了哪些环节？哪些算 AI Infra？

**参考答案**

**Q1：** 输入全部已知，所有 token 堆成矩阵一次乘完（并行）。输出第 N 个字依赖第 N-1 个字的结果，不知道上一个就算不出下一个（自回归），必须串行。

**Q2：** ① Q,K,V = X × Wq,Wk,Wv → ② scores = Q×Kᵀ → ③ scores + causal mask → ④ softmax → ⑤ weights×V。与此同时 K,V 顺手存进 KV Cache。Q/K/V 都是从同一个 X 变换出来的向量，乘了不同的 W，携带不同信息角度：Q 擅长提问、K 擅长自我介绍、V 擅长携带内容。

**Q3：** 不是。Prefill 跑完整 forward——每层先 Attention（顺手存 K/V）再过 FFN，缺一不可。只是 Attention 阶段顺手存了 K/V 供后续 Decode 复用。

**Q4：** 追加（append），不是重写。每步每层追加 1 行新 K/V。Decode 叫 memory-bound 因为每一步要读全部 KV Cache（可能几十 GB），但只算 1 个 token 的矩阵乘，大部分时间花在显存搬运数据上。

**Q5：** Attention 让 token 之间互相通信（谁跟我有关），FFN 让每个 token 独立做密集计算（加工 Attention 收集来的信息）。MoE 替换的是 FFN——把 FFN 拆成多个专家，路由选择激活。

**Q6：** 权重是训练产出的浮点数（如 0.0312、-1.847），几亿到几万亿个，按固定顺序排列。不是训练数据——训练数据是喂给训练程序的原始文本，不进模型文件。文件大是因为参数多，不是因为塞进了语料。

**Q7：** 预训练（猜下一个字→反向传播微调，✅错了往对调）、SFT（模仿人工标注问答对，✅有标准答案）、偏好对齐（人类偏好排序，⚠️不是对错是风格）、推理增强（可验证奖励自学推理，⚠️无标准答案自己试）。

**Q8：** 权重（常驻，最大）和 KV Cache（随对话增长）。GPU 几千核心并行矩阵运算，带宽 TB/s 级；CPU 几十核心，经 PCIe 搬运太慢。权重必须常驻 GPU 显存。

**Q9：** 推理部署工程师管「把训好的权重 load 起来、跑起来」；AI 算法工程师管「训练出权重」。你适合前者。

**Q10：** 客户端拼 Prompt（不是）→ 鉴权/限流（不是）→ 分词（是）→ Prefill(是) → Decode 循环(是) → 采样(是) → 流式返回。跟模型计算直接相关的算 AI Infra。
