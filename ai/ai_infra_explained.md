================================================================================
  从你已知的世界出发：AI Infra 全链路拆解
================================================================================

你现在对 AI 的认知是：打开网页 → 打字问问题 → AI 回复。
这就够了。下面从这个起点，把你不知道的中间部分全部拆开。


================================================================================
  第一节：一条聊天消息到底经过了什么
================================================================================

你在网页上给 ChatGPT 发一句话："今天天气怎么样"

当你点了发送键，到你看到回复，中间经历了这些：

  你的浏览器                                         ← 不是 AI Infra
    │  (把文本打包成 HTTP 请求，发到 ChatGPT 服务器)
    │
    ▼
  ChatGPT 后端服务器
    │  ── 通用后端逻辑（所有网站都有）：
    │      检查你是谁、有没有额度、有没有被封
    │      这部分不是 AI Infra，只是普通 Web 后台
    │
    │  ═══════════════ AI Infra 管辖范围 — 从这里开始 ═══════════════
    │
    │  ── 分词 (Tokenize)
    │      把"今天天气怎么样"切成 token 编号。
    │      大模型不认识汉字，只认数字编号。每个词/字对应一个数字。
    │      比如："今天"→[1234], "天气"→[5678], "怎么样"→[9012]
    │
    │  ── 拼装 Prompt
    │      把系统提示词 + 历史对话 + 你的问题拼成完整 token 序列
    │
    │  ── 转发给推理服务器
    │
    ▼
  推理服务器 —— 一台插着 GPU 的机器                  ← AI Infra 核心
    │  ── 加载模型：推理引擎从硬盘读几百 GB 模型文件到 GPU 显存
    │  ── 推理计算：token 序列 → N 层矩阵乘法 → 输出下一个最可能的 token
    │      每层是几十亿次乘加操作，全靠 GPU 并行运算
    │  ── 循环生成（自回归）：
    │      拿到新 token → 拼到末尾 → 再跑 → 再得 → 循环
    │      过程中需要"记住已生成的全部内容"，这个记忆机制叫 KV Cache
    │      KV Cache 是 LLM 推理中最吃显存的东西，占用几十到几百 GB
    │  ── 解码 (Detokenize)：
    │      把输出的 token 编号还原成人能读的汉字
    │
  生成的汉字逐字传回浏览器（你看到字一个一个蹦出来）

一句话总结：
  你只看到了打字和收到回复。中间的分词→拼Prompt→推理引擎→GPU→模型→KV Cache→解码
  ——这一整段就是 AI Infra 的管辖范围。
  鉴权/限流/计费是通用后端，所有网站都有，不属于 AI Infra。

关键概念——Token：
  大模型不读文字，读数字编号。你的输入先"分词"成一串编号，模型输出也是一串编号，
  最后再"解码"回文字。你看到的"一个字一个字蹦出来"，实际是"一个 token 一个 token 解码出来"。
  Token 是 AI Infra 中最基本的计量单位：模型收费按 token 数算，生成速度按 tokens/s 算，
  显存占用跟 token 序列长度直接相关。


================================================================================
  第二节：不同 AI 工具链路上的差异 —— ChatGPT / OpenCode / Cursor
================================================================================

你可能同时用 ChatGPT 网页版、OpenCode 终端、Cursor IDE 来问问题。
从 AI Infra 的视角看，它们的链路有本质区别。理解了这一点，
你就能分清每种工具里谁是"负责推理的"、谁是"负责做工具的"。

----------------------------------------------------------------
  工具 1：ChatGPT 网页版 —— "全托管"
----------------------------------------------------------------

  你的浏览器
    │  HTTP 请求
    ▼
  OpenAI 后端 ── 鉴权/限流/计费 ── Tokenize ── 拼 Prompt ──
    │
    ▼
  OpenAI 推理服务器（插满 GPU 的集群，跑着 GPT-4/GPT-5 模型）
    │
    ▼
  逐字返回结果到你的浏览器

  → AI Infra 归 OpenAI 管，模型、GPU、推理引擎全是 OpenAI 的。
    你不碰 AI Infra 任何东西，你只是消费者。

----------------------------------------------------------------
  工具 2：OpenCode 终端 —— "客户端 + 工具链"
----------------------------------------------------------------

  你的电脑
    │  OpenCode CLI 在你本机运行 —— 这是个客户端程序
    │  ── 项目管理：读你的代码库、跑 shell 命令、管理文件
    │  ── 工具编排：决定什么时候调什么工具（读文件、git、lsp 等）
    │  ── Prompt 拼装：把你的问题 + 代码上下文 + 工具结果拼成最终 request
    │
    │  HTTP/SSE 请求（发到 Anthropic/DeepSeek/OpenAI 的服务器）
    ▼
  API 网关（Anthropic/DeepSeek 的服务器）
    │  ── 鉴权、计费、限流
    │
    ▼
  推理服务器（Anthropic 的 Claude / DeepSeek 的模型）
    │  ── 流程和第一节一样：Tokenize → 推理 → KV Cache → 生成
    │
    ▼
  OpenCode CLI 收到回复 → 解析工具调用 → 执行工具 → 显示结果给你

  → AI Infra 部分（推理服务器、GPU、模型、分词/解码）是远程的、别人管。
    你本地跑的是客户端 + 工具链（opencode CLI），不是 AI Infra。
    opencode 的开发团队是"做工具"的人，不是"做推理"的人。

----------------------------------------------------------------
  工具 3：Cursor IDE —— "IDE + 模型接入"
----------------------------------------------------------------

  你的电脑
    │  Cursor 客户端（Electron App，VSCode 改的）在本机运行
    │  ── 代码索引（语法树、LSP）
    │  ── 上下文收集（打开的文件、光标位置、选中代码）
    │  ── Prompt 构造（系统提示 + 收集到的上下文 → 拼成请求）
    │
    │  HTTP 请求
    ▼
  Cursor 的后端服务器（Cursor 公司自己的）
    │  ── 鉴权/计费/订阅管理
    │  ── 模型路由：根据用户的订阅等级，决定调哪个模型、哪家 API
    │
    ▼
  推理 API（Anthropic / OpenAI 的推理服务器）
    │  ── 跟第一节一样：Tokenize → 推理 → 生成 → 返回
    │
    ▼
  Cursor 客户端收到 → 展示在编辑器里（内联补全 / Chat 面板）

  → Cursor 公司自己不跑模型推理，他们调 Anthropic/OpenAI 的 API。
    Cursor 的开发团队做的是：代码索引、上下文收集、UX 体验——这叫"AI 编程工具开发"，
    不是 AI Infra。

----------------------------------------------------------------
  三条链路的 AI Infra 边界对比（一张表看懂）
----------------------------------------------------------------

  ┌──────────────┬───────────────────┬───────────────────┬───────────────────┐
  │              │  ChatGPT 网页版    │  OpenCode           │  Cursor             │
  ├──────────────┼───────────────────┼───────────────────┼───────────────────┤
  │ AI Infra 在哪 │ 全在 OpenAI 机房   │ 远程（API 提供商） │ 远程（API 提供商）  │
  │ 你本地跑什么  │ 浏览器             │ OpenCode CLI+工具链  │ Cursor IDE+代码索引  │
  │ 你碰 AI Infra │ 完全不碰            │ 完全不碰            │ 完全不碰             │
  │ 开发商        │ 做推理+做产品      │ 做产品，调别人推理  │ 做产品，调别人推理   │
  └──────────────┴───────────────────┴───────────────────┴───────────────────┘

理解了这张表，你就不会被招聘 JD 上的称呼搞混：
  - 做 OpenCode/Cursor 这类工具的 → "AI 编程工具开发"或"AI Application 开发"
  - 做推理服务器、模型部署、GPU 算力管理的 → "AI Infra"
  - 两者完全不是一回事。

================================================================================
  第三节：把整条链路拆成零件（一个一个讲）
================================================================================

整条链路有五个关键零件。从你接触不到的 AI Infra 核心开始讲：

----------------------------------------------------------------
  零件 1：模型文件 —— 就是"大脑的脑浆"
----------------------------------------------------------------

一个模型文件就是一个超大文件，从几百 MB 到几百 GB。不是可执行程序，
就是一堆浮点数，存在硬盘上。

类比：你手机里下载了一本电子书（PDF/EPUB）。这本书只是一堆数据，
没有一个叫"阅读器"的程序打开它，你自己看不了。模型文件也一样——
没有一个叫"推理引擎"的程序打开它，它什么也干不了，就是一堆数字。

模型文件怎么来的？是算法工程师用大量数据训练出来的。
训练过程和你的工作完全无关，你不需要懂。

你需要知道的是：网上有无数现成的模型文件可以直接下载使用——不用自己训。

一个细节：有时候别人说的"模型"可能指的是"模型架构"（这张网络长什么样、
有几层、每层多大），也可能指的是"模型文件"（架构+训练好的参数，存成一个文件）。
日常沟通混着用，你心里分得开就行。

----------------------------------------------------------------
  零件 2：推理引擎 —— 就是"翻书的人"
----------------------------------------------------------------

推理引擎是一个程序，它的工作只有一件：打开那个模型文件，跑计算。

什么计算？输入一串文字 → 经过神经网络 N 层矩阵乘法 → 输出结果。
每一层都是在做"矩阵 × 矩阵"的数学运算，动辄几十亿次乘加操作。

类比：你下载了一本电子书（模型文件），但你需要一个阅读器 App（推理引擎）
来打开它、翻页、显示文字。你没有阅读器，那本书就是一块废数据。
你没有推理引擎，模型文件也只是一块硬盘上的数字，出不来回复。

你听过的 ONNX Runtime 就是一个推理引擎。它不是服务，不是网站，
就是一个程序库（C 语言写的），你调用它的 API 就能：
  - 加载模型文件
  - 喂给它输入
  - 拿到输出结果

ONNX Runtime 只是推理引擎中的一种。其他还有 TensorRT（NVIDIA 的）、
OpenVINO（Intel 的）、CoreML（Apple 的），原理都是一回事——加载模型、
执行计算。你入门用 ONNX Runtime 就够了：开源、支持纯 CPU、有 C API，
不需要你有 GPU。

----------------------------------------------------------------
  零件 3：推理框架 —— 就是"阅读器 Plus"
----------------------------------------------------------------

推理框架 = 推理引擎（基础版阅读器） + 更多高级功能。

基础版阅读器只能翻页、显示文字。Plus 版还能做笔记、高亮、翻译、查词。
推理框架也是这样——比裸推理引擎多出内存管理、请求队列、多卡并行等功能。

典型的推理框架：vLLM、SGLang、TensorRT-LLM。
它们底层调用推理引擎，上层提供更高级的能力。

对大模型（ChatGPT 这种）来说，裸推理引擎跑起来有一个致命问题：
大模型在逐字生成回复的过程中，需要记住"我之前已经生成的每一个字"，
否则后面的字就没有上下文。这个"记忆"占用的内存很大（几十到几百 GB），
而且每生成一个新字就要更新一遍。裸推理引擎需要你手动管这块内存，
稍不注意就爆显存或算错。vLLM 等框架专门把这块自动化了，
所以大模型场景下你不用裸引擎，而是用框架。

对你现在要做的事来说：先用 ONNX Runtime 这种轻量引擎入门就够了。
它的概念最干净，API 最简单，你不需要 GPU 就能跑，适合第一个项目。
大模型那一套（vLLM 等）后面再说。

----------------------------------------------------------------
  零件 4：推理服务器 —— 就是 7x24 小时运行的"客服"
----------------------------------------------------------------

推理服务器是一个一直在运行的网络程序。它绑在一个端口上，
等着别人从网上发请求过来。收到请求后，它内部调推理引擎跑计算，
算完了把结果返回给那个发请求的人。

一个推理服务器做的事列表：
  - 有个 URL 地址，别人可以访问
  - 接收请求：谁问的、问的什么
  - 检查权限、限流（一分钟只能问 60 次）
  - 把问题转给推理引擎
  - 把结果打包好返回
  - 记录日志：谁在什么时候问了什么问题、花了多少毫秒得出结果

类比：你家楼下的便利店，每天早上 7 点开门、晚上 11 点关门。
你推门进去说要一瓶水，店员拿给你。推理服务器就是那个 24 小时在线的店员，
推理引擎就是后面仓库里的存货和货架。

ChatGPT 就是一个推理服务器（极其巨大、极其复杂的一个）。

注意：推理引擎 不是 推理服务器。引擎只管"跑计算"（就像货架和水），
服务器要管"开门迎客"（收钱、记账、开门关门）。两个东西经常被弄混，
你从这里开始记住他们的区别。

----------------------------------------------------------------
  零件 5：客户端 —— 就是你现在用的东西
----------------------------------------------------------------

你问问题的那个界面（浏览器网页、OpenCode 终端、手机 App）就是客户端。

客户端的工作：让你打字 → 把你的字传给推理服务器 → 收到回复 → 显示给你。

类比：外卖 App 是"餐厅"的客户端。你不在厨房里做菜，你在 App 上下单。
浏览器是 ChatGPT 的客户端。你不在 GPU 上跑矩阵乘法，你在网页上打字。

----------------------------------------------------------------
  五个零件归属总览
----------------------------------------------------------------

  回到第一节的链路，把五个零件放回去，就能看清谁属于 AI Infra、谁不属于：

  ┌─────────────┬──────────────────────┬──────────────────────┐
  │ 零件          │ 是 AI Infra 吗？       │ 谁来管？               │
  ├─────────────┼──────────────────────┼──────────────────────┤
  │ 模型文件     │ ✅ 是                  │ 算法团队训练出来       │
  │ 推理引擎     │ ✅ 是                  │ AI Infra 团队部署/优化  │
  │ 推理框架     │ ✅ 是                  │ AI Infra 团队部署/优化  │
  │ 推理服务器   │ ✅ 是                  │ AI Infra 团队运维       │
  │ 客户端       │ ❌ 不是                │ 前端/App 团队          │
  └─────────────┴──────────────────────┴──────────────────────┘

  另外：鉴权/限流/计费（后端服务器那层）也不属于 AI Infra，是通用后端。
  分词/拼Prompt/解码 —— 属于 AI Infra，因为直接跟模型交互。

----------------------------------------------------------------
  一个关键配角：KV Cache（为什么大模型推理那么费显存？）
----------------------------------------------------------------

  大模型生成回复时是"自回归"的：先生成第1个token → 把第1个拼上 → 再生成第2个 →
  拼上 → 再生成第3个…… 每个新 token 都需要看到"前面已经生成了什么"才有上下文。

  如果每生成一个新 token 都重新算一遍前面所有的，那就太慢了。
  所以推理引擎会缓存前面层的中间计算结果——这个缓存就叫 KV Cache。

  KV Cache 有多大？跟已生成的 token 数量成正比。生成几百个 token 后，
  仅 KV Cache 就可能占几十到几百 GB 显存。这就是为什么：
  - 大模型很吃显存（不光是模型参数，KV Cache 更吃）
  - 推理框架（vLLM 等）核心功能之一就是高效管理 KV Cache
  - 长对话越聊越慢？因为 KV Cache 越来越大

  不需要背这个，知道"KV Cache = 大模型逐字生成时的记忆本，是大模型推理吃显存的首要原因"
  就行了。面试问到为什么 vLLM 比裸引擎强，你能提到 KV Cache 管理，已经比大部分人强了。

================================================================================
  第四节：AI Infra 各工种到底是干什么的（对号入座）
================================================================================

上面讲完了链路上的五个零件。接下来把这些零件映射到真实的招聘岗位——
你搜 JD 的时候看到这些名字就知道它对口的是链路的哪一环。

  ┌──────────────────┬────────────────────────────────────────────────────┐
  │  工种              │  工作内容                                          │
  ├──────────────────┼────────────────────────────────────────────────────┤
  │  推理部署工程师    │  负责把模型文件装到推理服务器上跑起来：              │
  │                    │  - 搭运行环境（装推理引擎、装驱动）                 │
  │                    │  - 配置服务（端口、限流、日志）                     │
  │                    │  - 上线后盯着：QPS 够不够？延迟有没有毛刺？         │
  │                    │    报错了怎么定位？                                │
  ├──────────────────┼────────────────────────────────────────────────────┤
  │  ML Platform      │  开发/维护一个"模型上线流水线"的平台：               │
  │  工程师            │  让算法工程师能自助把模型一键部署上线，             │
  │                    │  而不需要每次找推理部署工程师手动操作。             │
  │                    │  还包括：模型版本管理、自动测试、资源调度。          │
  ├──────────────────┼────────────────────────────────────────────────────┤
  │  MLOps /          │  负责模型从开发完成到上线的全流程自动化。             │
  │  AI DevOps        │  举个例子：当算法团队训练好一个新版本模型，          │
  │                    │  需要把它部署到线上替换旧版本。这个"替换"过程        │
  │                    │  包括：自动跑一遍测试确保新模型不会把服务搞崩、       │
  │                    │  把模型文件推到所有服务器上、切换流量、              │
  │                    │  如果上线后报错率飙升就自动滚回旧版本。              │
  │                    │  MLOps 工程师就是设计并维护这条自动化流水线的人。    │
  ├──────────────────┼────────────────────────────────────────────────────┤
  │  推理优化工程师    │  深入到推理引擎内部，优化它跑计算的速度：            │
  │                    │  - 写 GPU kernel（CUDA 代码）                       │
  │                    │  - 算子融合（把多个矩阵运算合成一个）               │
  │                    │  - 需要 GPU 和 C++ 深度功底                        │
  ├──────────────────┼────────────────────────────────────────────────────┤
  │  AI 算法工程师     │  训练模型、设计模型架构、处理训练数据、调参。        │
  │                    │  这是造"大脑"的人 —— 跟 AI Infra 完全无关。         │
  └──────────────────┴────────────────────────────────────────────────────┘

你应该投前三行：推理部署、ML Platform、MLOps。
这三行吃的是你"能排查、能闭环、能交付"的能力，不需要你写模型算法。


================================================================================
  第五节：你的业余项目到底做什么（为什么做）
================================================================================

你不需要搭一个完整的推理服务器。你不需要 7x24 小时运行任何东西。
你只需要做一件事：写一个 C 程序，调用 ONNX Runtime 的 C API，
加载一个现成的模型文件，跑推理，记录性能数据。

具体步骤：

  步骤 1：下载一个模型文件（现成的，不自己训练）
          比如从网上下载 ResNet-18 的 ONNX 版本，大约 45MB

  步骤 2：写一个 C 程序，大约 200-300 行
          核心逻辑就四件事：
          ① 调用 ONNX Runtime 的 C API 加载那个 45MB 的模型文件
          ② 准备输入（一张猫的图片，转成模型需要的数字格式）
          ③ 调 Run() 函数跑一次推理——返回值是"这是猫，置信度 95%"
          ④ 同样的步骤循环跑 1000 次，记录每次的耗时

  步骤 3：输出一份 benchmark 报告
          内容包括：
          - 跑了什么模型（ResNet-18）
          - 在什么设备上（你的 CPU 型号）
          - 每次跑一张图：每秒能跑多少次、平均延迟多少 ms、
            最慢的 1% 是什么情况

参考答案速查（别偷看，先自己答一遍）：

Q1: 浏览器(不是AI Infra) → 鉴权/限流(不是AI Infra，通用后端) → 分词(是AI Infra)
    → 推理引擎+GPU(是AI Infra核心) → KV Cache管理(是AI Infra) → 解码(是AI Infra)
    → 返回浏览器。记住分界线：跟模型计算直接相关的算AI Infra，鉴权/计费/限流不算。

Q2: Token是大模型的最小处理单元，不是完整的字。中文里一个token可能是一个字或一个词。
    大模型只认识数字编号，所以输入要分词(tokenize)成编号序列，输出也是编号序列，
    再解码(detokenize)回汉字。你看到的"蹦字"实际是"蹦token"。

Q3: ChatGPT是全托管(AI Infra+产品都在OpenAI)；OpenCode/Cursor只做客户端+工具链，
    推理调的是Anthropic/OpenAI/DeepSeek的API。OpenCode和Cursor的开发团队是"做工具"的，
    不是"做推理"的。

Q4: ONNX Runtime是推理引擎(零件2)，负责加载模型、执行矩阵计算。不是服务器——
    服务器要绑定端口、接收网络请求；引擎只管"跑计算"。

Q5: 推理引擎是算东西的程序(加载模型→跑矩阵乘法→输出结果)；
    推理服务器是7x24在线、绑定端口、接收网络请求的服务程序。

Q6: 大模型逐字生成时需要"记住已生成的每个字"（KV Cache），这个记忆占用几十到几百GB显存，
    需要专门的内存管理。裸引擎不会自动管这个，推理框架(vLLM等)把它自动化了。

Q7: 你的业余项目是"本地跑一个小模型做benchmark"，加载一次模型、跑一次推理、记录耗时。
    ChatGPT是"几百GB大模型+GPU集群+7x24小时在线服务"。你不需要GPU因为小模型CPU就能跑。

Q8: 推理部署工程师管"把模型跑起来"(装引擎、配服务、盯QPS)；
    AI算法工程师管"造模型"(训练、调参)。你适合推理部署——吃工程闭环能力，不需要写模型算法。
