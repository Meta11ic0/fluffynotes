### 1.1  你今天用的 AI，底层到底是什么

你用 ChatGPT 问问题、用 Claude 分析文档——这些产品背后，跑的是同一个东西：**大语言模型（LLM，Large Language Model）**。如果你也用 Cursor 写代码，它底层也是同一个原理，只是多了一层工具调用。

如果只用一句话概括今天的大模型 AI：

**它是一个在海量文本上训练出来的「下一个字预测器」，但因为规模足够大、训练足够充分、外围工程系统足够完善，表现出来像一个能理解语言、调用工具、完成复杂任务的数字助手。**

注意这里有三个层次：

1. **底层本质**：它不是在「思考」，而是在做概率预测——给定前面的字，猜下一个字最可能是什么。
2. **表层体验**：因为对语言、代码、知识都训练得足够多，你感觉它像一个「会交流、会做事的人」。
3. **工程现实**：它之所以能在产品里可用，不只是模型强，还因为外面有一整套推理系统、缓存系统、调度系统、观测系统在支撑。

你现在用 Cursor 的体验，本质是：

```
你输入自然语言
  → 模型理解上下文（代码文件、历史对话、项目结构）
  → 系统决定调用哪些工具（读文件、改文件、执行命令）
  → 工具返回结果
  → 模型根据结果继续规划或回答
```

所以你的感受——「AI 像一个大型的人，我给她清晰指令，她结合操作库匹配动作」——**在产品体验层面是对的**。但在技术层面，更准确的说法是：

**模型像「大脑」（推理引擎），工具像「手和外部器官」，基础设施像「神经系统和后勤系统」。**

### 1.2  AI → ML → DL：三层关系一张图

这三个词经常混用，但它们不是同一层级。

```
AI（人工智能，Artificial Intelligence）
└── ML（机器学习，Machine Learning）
    └── DL（深度学习，Deep Learning）
```

#### AI（人工智能）——最外层，最宽泛

让机器表现出「像智能」的行为。导航推荐路径、短视频推荐、语音助手、聊天机器人——这些都算 AI。AI 是**总目标**，不是一种具体算法。

#### ML（机器学习）——AI 的实现路径之一

核心思想：**不手写每条规则，让机器从数据里学习规律。**

举个例子（垃圾邮件检测）：
- **规则系统**：你手写「含'中奖'就是垃圾邮件」
- **ML 系统**：给模型 10 万封带标签邮件，它自己学「哪些模式更像垃圾邮件」

ML 的关键效果：遇到没见过的表达也能判断对，可扩展性远高于手写规则。

#### DL（深度学习）——ML 中最成功的一大类

用**多层神经网络**从数据中学习。你在 1.1 读到的大语言模型（LLM），底层就是一个规模极大的深度神经网络——几十到上百层，每层做固定的数学运算，层与层之间传递结果。对图像、语音、文本这类复杂模式识别效果特别好，在大数据和大算力条件下性能提升远超传统 ML 方法。

**一句话关系：** AI 是目标，ML 是路径，DL 是当前最成功的那条路。

### 1.3  纠正一个关键误解：AI ≠ if-else + 数据库

这是一个非常常见、也非常合理的初始理解：

> 「AI 是不是就是程序接一个数据库，用户问 A 就去数据库查相关答案，然后组织成自然语言回答？」

这个理解**有一部分对，但不完整**。把它彻底搞清楚，后面的学习才不会走偏。

#### 规则系统：人写规则

```
如果用户问"营业时间"，回答"9:00-18:00"
如果用户问"退货规则"，回答"7 天无理由"
如果用户说"查询订单"，调用订单接口
```

- 规则由人手工编写
- 规则覆盖到哪里，能力就到哪里
- 没写到的场景，系统就不会

#### 学习系统：机器自己学

真正的变化不是「把 if-else 换成数据库查询」，而是：**从「人手工写规则」变成「模型自己从数据里学出模式和表示」**。

| | 规则系统 | 学习系统 |
|---|---|---|
| **知识在哪** | 人写的规则里 | 模型的参数（数字）里 |
| **遇到没见过的情况** | 通常失效 | 可能泛化应对 |
| **扩展方式** | 加更多规则（越来越难维护） | 加更多数据（模型自己学） |

这里说的「参数」到底长什么样？不是配置文件里的几个设置项，而是**几亿到几万亿个浮点数**（比如 `0.0312`、`-1.847`），按固定顺序排列。这些数字不是人写的——是模型从训练数据里「练」出来的。你可以把它们想象成几万亿个微调旋钮，每个旋钮拧到什么刻度，决定了模型看到某种输入时更可能输出什么。训练的过程就是不断拧这些旋钮，让输出越来越接近期望答案。这在 3.1 节会展开。

那回到你最初的理解——「AI 就是接数据库查答案」。规则系统对应了你的"if-else"，而"接数据库"这部分有没有对应的东西？**有，它叫 RAG。**

#### 那 RAG（检索增强生成）是什么

**RAG = Retrieval-Augmented Generation = 检索增强生成。** 一句话：**先查资料，再回答。**

为什么需要 RAG？因为大模型有两个先天局限：

1. **知识截止在训练时**——训练完成后参数固定，不知道之后发生的事
2. **可能"幻觉"**——编造不存在的信息，尤其在需要精确事实的时候

RAG 的做法很直白：用户提问后，先从外部知识库检索相关文档片段，把这些片段连同问题一起喂给模型，让模型**基于资料**组织回答，而不是纯靠记忆。

```
没有 RAG：    用户问 → 模型凭记忆答（可能过时、可能编造）
有 RAG：      用户问 → 检索知识库 → 找到相关文档 → 拼进 Prompt → 模型基于资料答
```

**规则系统和学习系统是两种根本不同的范式**——前者是人写逻辑，后者是从数据中学。**RAG 不是第三种范式，而是叠加在学习系统之上的一个增强技术**——给大模型接一个外部资料库，弥补它记忆有限、知识过时的问题。

用类比来感受它们的关系：

```
规则系统：        像一本写死流程的操作手册
                  没写到的就不会，但写到的绝对按规矩来

学习系统（大模型）：像一个读过很多书的人，凭记忆回答
                  泛化能力强，但知识截止在训练时，可能记错

     └─ 大模型 + RAG：给这个人配了一个随时可以翻的资料库
                      他仍然用自己的语言能力组织回答
                      但遇到需要精确事实的时候，先翻资料再说话
```

RAG 不是大模型的替代品——它的底座仍然是大模型。没有大模型的语言能力，光有资料库什么也做不了。

> **现代 AI 产品的核心公式是 模型 + RAG + 工具调用。** 模型负责理解和生成，RAG 接外部知识弥补时效和幻觉问题，工具调用让模型不只"说"还能"做"（读文件、查数据库、调 API）。规则系统在需要绝对确定性的地方（安全过滤、格式校验）仍然有用，但它不是 AI 产品智能部分的主力——它是独立于模型之外的兜底逻辑。RAG 的具体能力和局限在 §3.3 展开。

### 1.4  一个句子是怎么被模型「理解」的

> **初读提示：** 本节用六步追踪一个真实句子的完整旅程——不是讲矩阵运算，而是建立直觉。如果只想先建立全局概念，可以跳到 §1.5「核心名词速览」，等读完第三章再回来看这一节。想了解每一层的精确矩阵计算和形状变化，见 [ai-engineering.md §2](ai-engineering.md)。

用「我 喜欢 吃 红烧肉」这句话，从输入到第一个输出字，追踪全过程。

#### 第一步：把文字变成编号

你打的汉字，模型一个都不认识。它只处理数字。所以第一件事是**分词**——用一个叫 Tokenizer 的工具把句子拆成小块（token），每个小块对应词表里的一个整数编号。

```
输入: "我喜欢吃红烧肉"
输出: 4 个整数，比如 [8247, 15302, 901, 43891]
```

一个 token 可能是一个完整的词（「红烧肉」），也可能是一个字甚至半个字——取决于分词器的设计。模型按 token 数量收费，生成速度按每秒 token 数（tokens/s）衡量。你看到字往外蹦，底层是一个一个 token 在往外蹦。

#### 第二步：把编号变成有意义的数字串

光有编号不够，模型还需要知道每个 token 的**语义**。于是每个编号被映射成一个高维向量——一长串小数。这个过程叫 Embedding。

关键特性：意思相近的词，向量在数学空间里的距离也近。「猫」和「狗」的向量很近，「猫」和「电脑」的向量很远。这个性质不是人设定的——是模型在海量文本中自己学会的。

#### 第三步：告诉模型"谁在前谁在后"

现在每个 token 都有了一个携带语义的向量。但有一个盲区：模型还不知道顺序。「我 喜欢 吃 红烧肉」和「红烧肉 喜欢 吃 我」用的是同一组词的向量，含义完全不同，但只看向量的话它们长得一模一样。

解决方法是给每个位置贴一个标签——**位置编码**。第 1 位的「我」带上"位置 1"的标记，第 4 位的「红烧肉」带上"位置 4"的标记。这样模型就能区分"谁先出现、谁后出现"。

#### 第四步：Attention 计算「当前词该关注谁」

有了位置信息，模型现在知道每个词在句子的哪个位置。但处理「吃」这个词时，前面三个词「我」「喜欢」「红烧肉」跟它的关系并不一样——「红烧肉」是被吃的东西，显然比「我」更直接相关。

模型需要一种机制来判断「处理当前词时，前面哪些词更重要」。这个机制就叫 **Attention（注意力）**。

- 对「红烧肉」权重大（吃的东西）
- 对「喜欢」也有权重（动作关系）
- 对「我」有权重（谁在吃）

这就是 Attention 的核心直觉——**不是平均地看每个字，而是动态聚焦于当前最相关的部分**。

#### 第五步：多层堆叠，逐层抽象

一层 Attention 不够。实际模型会堆几十到上百层（比如 Llama-3-8B 有 32 层），每一层都在前一层的基础上继续加工。用一个稍复杂点的例子来看每一层"看到了什么"：

> 「小明说他昨天去了那家新开的川菜馆，菜很辣但是他很喜欢」

这个句子经过分词后大约有十几个 token。为了展示逐层编码的过程，我们追踪**最后一个 token「喜欢」**（或 tokenizer 切出的子词「欢」）——因为它在句末，经过因果注意力机制能看到前面所有 token，携带的上下文最完整。

每一层产出的不是自然语言"结论"，而是一组高维向量（一串数字）。但研究者做过大量**探针实验**——在每层输出的向量上训练一个简单分类器，看它能不能识别出某种信息——由此反推出各层向量里**编码了什么**。

下面用探针视角看「喜欢」这个位置的向量在各层编码了什么：

```
第 1-4 层（表层编码）：
  向量里能检测到的："喜欢"前面紧挨着"很"，再往前是"他"
  向量里检测不到的：这个"他"是谁、整句的结构如何

第 5-12 层（句法编码）：
  向量里能检测到的：句子由「小明说…」和「菜很辣但是他很喜欢」两个大块组成；"但是"标记了转折关系；"他"与句首的"小明"共享指代
  往前推进了：不再只是相邻字的关系，而是识别出了句子的骨架

第 13-24 层（语义编码）：
  向量里能检测到的："辣"和"喜欢"被"但是"隔开，二者的情绪方向相反——「辣是负面体验」和「喜欢是正面态度」构成矛盾
  往前推进了：不只是谁修饰谁，而是开始编码词与词之间的语义冲突

第 25-32 层（语用编码）：
  向量里能检测到的：矛盾被消解——"喜欢"被编码为句子的重心，"辣"被重新编码为正面信号（"因为辣所以好吃"）
  往前推进了：整体情绪是正面的，说话人的意图是推荐而非抱怨
```

**两点补充：**

第一，上面的例子只追踪了「喜欢」这一个位置，但实际上**输入句子的每一个 token 都在同时走这个过程**——「小」「明」「说」「他」「昨」「天」……每个字经过全部 32 层后，都得到了一个编码了丰富信息的向量。只是不同位置的向量携带的上下文量不同——越靠后的位置看到的越多。

第二，编码完之后呢？这就到了第六步。

#### 第六步：从向量到「下一个字」

32 层全部走完后，每个 token 都得到了一个编码了整句上下文的最终向量。要预测下一个字，模型取**最后一个 token** 的最终向量——在这个例子里，最后一个 token 是「喜欢」（或 tokenizer 切出的「欢」），不是第五步举例用的「他」。这个向量经过最后一轮矩阵乘法，被**投影到词表大小**（比如词表有 10 万个 token，就输出 10 万个分数），再经过 softmax 变成概率分布：

```
"喜欢"的最终向量 → [最后一层矩阵乘法] → ["吃": 0.31, "的": 0.18, "。" : 0.12, "这家": 0.08, ...]
```

然后按策略选一个——贪心就选「很」，加一点随机性可能选「也」。选出来的 token 追加到输入末尾，整个流程再跑一遍，生成下一个。这就是你在屏幕上看到的一个字一个字往外蹦。

**所以 LLM 的本质就是「下一个 token 预测器」——输入一串 token，经过几十层逐层编码，最后把末尾位置的向量投影成词表上的概率分布，挑一个。** 只是规模大到一定程度后，表现出惊人的通用能力。

### 1.5  核心名词速览

这一节不展开，只让你快速知道「这个词在系统里起什么作用」。需要深入理解时翻到第三章。

#### 基础概念

| 名词 | 一句话 | 在哪里用 |
|------|--------|---------|
| **LLM** | 大语言模型，在超大规模文本上训练出来的"下一个 token 预测器" | 所有 AI 聊天/编码产品 |
| **Token** | 模型处理文本的最小单位，不一定是完整的一个字 | 分词、计费、速度计算 |
| **Embedding** | 把词/句映射成数字向量的技术，语义接近的向量也接近 | 检索、相似度计算、RAG |
| **Transformer** | 2017 年提出的以 Attention 为核心的神经网络架构 | 今天所有主流大模型的底座 |
| **Attention** | 让每个 token 在处理时动态关注上下文中其他 token 的机制 | Transformer 核心组件 |

#### 模型生命周期

| 名词 | 一句话 | 谁在做 |
|------|--------|--------|
| **训练（Training）** | 在海量数据上学习参数，形成模型通用能力 | 算法团队 |
| **推理（Inference）** | 训练好的模型上线响应用户请求 | AI Infra / 推理工程师 |
| **微调（Fine-tuning）** | 在预训练模型基础上做任务定制 | 算法 + 工程 |

#### 从「会说」到「会干活」

| 名词 | 一句话 | 解决的问题 |
|------|--------|-----------|
| **RLHF** | 基于人类偏好反馈优化模型输出 | 让模型更"听话"，减少有害回答 |
| **RAG** | 先检索外部知识，再让模型基于检索结果生成答案 | 时效性、私有知识、可追溯性 |
| **Agent** | 模型 + 工具 + 规划执行流程的组合系统 | 不只回答问题，还能调用工具执行任务 |

#### 基础设施常见词

| 名词 | 一句话 | 你会在 JD 里看到 |
|------|--------|-----------------|
| **推理引擎** | 高效执行推理计算的核心组件（如 ONNX Runtime、TensorRT） | ✅ |
| **推理框架** | 引擎 + KV Cache 调度 + 多卡并行 + 请求队列（如 vLLM、SGLang） | ✅ |
| **KV Cache** | 缓存推理过程中的中间状态，避免重复计算 | ✅ 高频 |
| **GPU** | 在 AI 中的核心价值是并行矩阵运算，不是画图 | ✅ |
| **QPS** | 每秒可处理请求数，衡量系统容量 | ✅ |
| **P95/P99** | 95%/99% 请求在该时间内完成，衡量用户真实体验 | ✅ |
| **Batching** | 把多个请求合并一起算，提升吞吐 | ✅ |
| **FastAPI** | Python 异步 Web 框架，AI 服务化的事实标准 | ✅ 高频 |
| **Docker / K8s** | 容器化 + 编排，AI 部署必备 | ✅ 高频 |
| **SLM** | 小语言模型（0.5B-7B），生产环境中成本仅为大模型 1/50 | ✅ 增长中 |

> 以上各项的 2026 年真实数据：vLLM ~86k stars（40万+ GPU）、SGLang ~30k stars、ONNX Runtime ~21k stars / v1.27.0、FastAPI ~100k stars / 38% 开发者采用、K8s 82% 容器用户生产使用、SLM 92%+ HF 下载为 1B 以下。完整数据及来源链接见 [ai-engineering.md §5](ai-engineering.md)。

#### 全景图

**五层工作流**：用户请求依次穿过这五层，每层有自己在做的事和主流技术。

```
用户: "这段代码什么意思"
  │
  ▼
第5层 应用增强       RAG 查资料 / Agent 调工具 / LangChain 编排
  │  (可选)           补知识、补行动力
  ▼
第4层 部署交付       FastAPI 收请求 / SSE 逐字推流 / Docker 跑服务
  │                  把模型包成可访问的 API
  ▼
第3层 推理引擎/框架   vLLM 调度请求 / ONNX Runtime 执行矩阵计算
  │                  决定"怎么跑、跑多快"
  ▼
第2层 模型           Qwen / Llama / DeepSeek 的权重文件（几十GB）
  │                  "谁的知识在回答"
  ▼
第1层 模型架构       Attention 让 token 互看 / FFN 深度加工每个 token
  │                  决定"模型内部怎么算"
  ▼
输出 token → 逐字推回用户
```

**各层竞品**：前 3 层有真正的竞争关系（同一行 `、` 分隔 = 互替），后 2 层是工具链组合。

**第1层 — 模型架构**

Attention 和 FFN 各自有一个"目的"，不同方案是达成这个目的的不同实现：

| 目的 | 经典实现 | 后来的实现 |
|---|---|---|
| 让 token 之间互看（Attention） | 标准 Attention | MLA（DeepSeek V2/V3，压缩 K/V 省显存） |
| | | CSA+HCA（DeepSeek V4，两级压缩+稀疏索引） |
| | | GatedDeltaNet（Qwen 3.6，线性替换 softmax） |
| （辅助）加速、省显存 | — | FlashAttention（通用加速）、GQA（Llama 3，省显存） |
| 深度加工每个 token（FFN） | 标准 FFN | SwiGLU（Llama / Qwen / DeepSeek 主流） |
| | | MoE（DeepSeek V4 / Qwen3.6 / Llama 4，每次只激活部分专家） |

> Attention 和 FFN 各自独立迭代，一个模型可以同时选两边的任意方案（如 DeepSeek V4 = CSA+HCA + MoE）。

**第2层 — 模型**（权重文件）

| 开源 | 闭源 |
|---|---|
| Qwen、Llama、DeepSeek、Gemma、Mistral | GPT、Claude、Gemini |

**第3层 — 推理引擎/框架**（把模型跑起来）

| 层级 | 做什么 | 主流选择 |
|---|---|---|
| 推理框架（上层） | 调度请求、管 KV Cache、对外服务 | vLLM、SGLang、TensorRT-LLM、Ollama、llama.cpp |
| 推理引擎（底层） | 加载模型、执行矩阵计算 | ONNX Runtime、TensorRT、PyTorch 原生 |

> GPU 生产：vLLM + SGLang 协同。本地：Ollama。CPU/边缘设备：ONNX Runtime。

**第4-5层 — 部署交付 + 应用增强**

这两层没有竞品关系——各环节是串联的工具链，不是互替选项：
- 部署链：FastAPI（API）→ SSE（流式）→ Docker（容器）→ K8s（编排）→ Prometheus（监控）
- 增强链：RAG（LangChain / LlamaIndex + ChromaDB / Milvus）→ Agent（LangGraph / AutoGen）

## 第二章：AI 是怎么一步步发展到今天的

下面不是「所有历史事件」的堆砌，而是**对今天生态影响最大的关键转折点**。

读完这章你会理解：为什么 2022 年 ChatGPT 突然出现不是「偶然」，而是几十年算法、硬件、数据、工程的共同积累到了一个临界点。

---

### 2.1  第一阶段：AI 概念诞生，但产品不存在（1956-1985）

#### 1956：AI 这个词被正式提出

达特茅斯夏季研讨会上，John McCarthy 等人提出了「Artificial Intelligence（人工智能）」这个术语。

这件事的意义不是「突然做出 AI 了」，而是：**从那一刻开始，人们明确把「让机器模拟智能」当作一个独立学科来研究。**

> 参考：[IEEE 里程碑：1956 引入 AI 术语](https://ieeemilestones.ethw.org/Milestone-Proposal:Introduction_of_the_Term_%E2%80%98Artificial_Intelligence,%E2%80%99_1956)

#### 1957-1958：感知机（Perceptron）

Frank Rosenblatt 提出的感知机，是神经网络的早期原型。它第一次让「机器通过训练获得分类能力」有了较具体的数学和工程实现。

但它只能解决很简单的线性可分问题——能力远远不够支撑复杂智能。

> 参考：[Rosenblatt 1958 Perceptron 论文](https://www.ling.upenn.edu/courses/cogs501/Rosenblatt1958.pdf)

#### 这一阶段的普通人视角

这一时期的 AI 对普通人几乎**没有产品存在感**。原因很简单：数据不够、算力不够、算法不够成熟、工程体系也远不成熟。AI 更像科研概念，而不是大众产品。

---

### 2.2  第二阶段：神经网络复苏，GPU 入场（1986-2011）

#### 1986：反向传播（Backpropagation）成为关键训练方法

Rumelhart、Hinton、Williams 等人让反向传播成为多层神经网络的标准训练方法。

它解决的问题是：**多层神经网络怎么训练。** 没有反向传播，深层网络无法有效学习——模型输出和正确答案的误差，通过这个算法逐层往回传，精确修正每个参数。

> 参考：[Rumelhart / Hinton / Williams 1986, Nature](https://www.nature.com/articles/323533a0)

#### 1989-1998：LeNet 和卷积神经网络的早期实践

Yann LeCun 提出的 LeNet 在手写数字识别（MNIST）上取得成功，证明了卷积神经网络（CNN）在图像任务上的潜力。这是深度学习在工业场景的早期探索。

#### 1997：LSTM 解决长序列问题

LSTM（Long Short-Term Memory，长短期记忆网络）解决了普通 RNN（Recurrent Neural Network，循环神经网络——当时处理序列数据的标准模型）的一个致命问题：「梯度消失」——处理长序列时，早期的信息会在逐层传递中变得越来越微弱，最终几乎消失。LSTM 通过精巧的门控机制截断了这个衰减过程，让**机器处理长序列信息**第一次变得真正可行，在语音、翻译、时间序列任务中影响深远。

> 参考：[LSTM 原论文](https://direct.mit.edu/neco/article-abstract/9/8/1735/6109/Long-Short-Term-Memory)

#### 2006：CUDA 出现——GPU 不再是显卡，而是通用并行计算平台

这一点非常关键，但普通人常常忽略。

NVIDIA 发布 CUDA，让开发者可以用 C 语言直接对 GPU 编程。这意味着 GPU 不再只是画图的，而成为了**通用并行计算平台**。

为什么这会改变 AI？因为神经网络训练本质上是海量矩阵运算——几百万个数字同时做乘法和加法。CPU 有几十个计算核心，擅长「一件件快速做」；GPU 有几千个计算核心，擅长「几千件事同时做」。矩阵运算天生适合后者。后来的深度学习爆发，不只是算法变好了，也是因为**终于有一类硬件特别适合跑它**。

> 参考：[NVIDIA 2006 CUDA 介绍](https://www.cgw.com/Press-Center/News/2006/Nvidia-Introduces-CUDA-Architecture-for-Computin.aspx)

#### 这阶段的开发者视角

在这个阶段，AI 生态真正发生的变化是：规则系统不再是唯一方向，数据驱动方法开始占上风，算法和硬件逐渐形成「正反馈」循环。但这时还没有今天的大模型。

---

### 2.3  第三阶段：深度学习爆发——AlexNet 改变生态（2012-2016）

#### 2012：AlexNet——深度学习爆发的发令枪

AlexNet 在 ImageNet 图像分类竞赛上以巨大优势夺冠（错误率从 26% 降到 15%）。它的重要性在于**三件事同时成立**：

1. 深层网络真的能赢
2. GPU 训练真的可行
3. 大数据集 + 深网络 + GPU 算力这个路线被证明有效

虽然普通人那时还没直接感知，但在产业界和学术界，这个事件几乎像一记发令枪。

> 参考：[AlexNet 论文](https://proceedings.neurips.cc/paper_files/paper/2012/file/c399862d3b9d6b76c8436e924a68c45b-Paper.pdf)

#### 2013：word2vec——让「词向量」流行起来

word2vec 把词表示成向量，让机器能更好理解「语义相近」的关系。这是 NLP（Natural Language Processing，自然语言处理——让机器理解和生成人类语言的技术领域）从传统人工特征工程走向神经方法的重要一步。

> 参考：[word2vec 论文](https://arxiv.org/abs/1301.3781)

#### 2014：seq2seq + Attention

- **seq2seq**（Sequence to Sequence）：输入一段序列，输出另一段序列——「端到端文本生成」范式的里程碑。典型场景：机器翻译。
- **Bahdanau Attention**：模型在生成每个词时，动态关注输入中的不同部分。这是后面 Transformer 的直接前身。

> 参考：[seq2seq 论文](https://papers.nips.cc/paper/5346-sequence-to-sequence-learning-with-neural-networks) | [Bahdanau Attention 论文](https://arxiv.org/abs/1409.0473v7)

#### 2016：AlphaGo 进入大众视野

虽然 AlphaGo 不是语言模型，但它对公众认知的影响巨大。普通人第一次强烈感受到：**原来 AI 已经不是实验室玩具，而是能在复杂领域击败顶尖人类。**

> 参考：[AlphaGo Nature 论文](http://www.nature.com/articles/nature16961.pdf)

---

### 2.4  第四阶段：Transformer 登场——大模型的底座（2017-2019）

#### 2017：Transformer——真正的大拐点

Google 发表论文《Attention Is All You Need》，提出 Transformer 架构。

为什么它如此重要？因为它解决了以前 RNN/LSTM 体系的核心瓶颈：

| RNN/LSTM 的问题 | Transformer 的方案 |
|---|---|
| 必须按顺序一步步处理，无法高效并行训练 | 序列中各位置可同时处理，充分利用 GPU 并行能力 |
| 长序列中早期信息容易丢失 | Attention（回顾 1.4 第四步——每个 token 动态关注上下文中所有其他 token）让每个位置直接连接所有其他位置，不受距离限制 |
| 扩展到海量数据和超大模型受限 | 天生适合大规模扩展 |

**Transformer 让「更大规模的语言建模」从学术实验变成了工程现实。** 它是今天 GPT、Claude、Gemini、Llama、DeepSeek、Qwen 所有主流模型的技术底座。

> 参考：[Transformer 原论文](https://proceedings.neurips.cc/paper_files/paper/2017/file/3f5ee243547dee91fbd053c1c4a845aa-Paper.pdf)

#### 2018：BERT——预训练 + 微调范式

BERT（Bidirectional Encoder Representations from Transformers）让 NLP 任务取得全面大幅提升。它代表的是**预训练 + 微调**范式的大成功：

- 先在大规模语料上通用预训练
- 再在具体任务上稍作微调

这意味着**不需要为每个 NLP 任务都从零开始训练一个模型**。

> 参考：[BERT 论文](https://arxiv.org/abs/1810.04805v1)

#### 2019：T5 与「The Bitter Lesson」

- **T5**（Text-To-Text Transfer Transformer）：把几乎所有 NLP 任务统一成「输入文本 → 输出文本」的形式。这对后来的大模型产品很重要——今天你看到的聊天、翻译、摘要、代码补全，本质上都在走「统一接口」这条路。
- **The Bitter Lesson**：Rich Sutton 的文章在思想层面影响巨大。核心观点：**长期看，依赖通用方法 + 大规模计算，通常比大量手工规则和领域技巧更有效。** 这句话直接预言了大模型时代的到来。

> 参考：[T5 论文](https://arxiv.org/abs/1910.10683v3) | [The Bitter Lesson](https://bitterlesson.ai/)

---

### 2.5  第五阶段：GPT-3 → ChatGPT——大众第一次感知 AI（2020-2022）

#### 2020：GPT-3——「通用能力」第一次震撼行业

GPT-3（Generative Pre-trained Transformer 3）最大的行业意义不只是参数大（1750 亿），而是：

**大家开始真切看到：只要模型够大，很多任务不一定要专门训练，也能「像会做一样」完成。**

- 少样本学习（Few-shot）：给它几个例子，它就能举一反三
- 一个模型同时做翻译、问答、写作、摘要、代码生成
- 产业界意识到：语言模型不再只是 NLP 学术工具，而可能成为**通用接口**

> 参考：[GPT-3 论文](http://arxiv.org/abs/2005.14165)

#### 2020：RAG——先查资料，再回答

RAG（Retrieval-Augmented Generation，检索增强生成）的出现，是因为大家很快发现大模型的两个问题：
1. 知识停在训练时，不知道最新事实
2. 可能会「幻觉」——编造不存在的信息

RAG 的核心思想很简单：**先查资料，再回答。** 大模型像「语言能力很强的人」，知识库像「可查阅的外部资料」，RAG 就是让它回答前先翻资料。

> 参考：[RAG 论文](https://arxiv.org/abs/2005.11401)

#### 2021：Foundation Model 概念 + LoRA 微调

- **Foundation Model（基础模型）** 概念被正式提出：先训练一个通用大模型，再在很多具体场景上适配。模型开始像「底层平台」，下游应用不必每次重来。
- **LoRA**（Low-Rank Adaptation）：低成本微调方法。不是所有场景都要重新全量训练模型，很多时候可以更便宜地定制。

> 参考：[Foundation Models 报告](https://arxiv.org/abs/2108.07258) | [LoRA 论文](http://arxiv.org/pdf/2106.09685)

#### 2022：InstructGPT + RLHF → ChatGPT

GPT-3 很强，但并不一定「听话」。InstructGPT 引入 RLHF（Reinforcement Learning from Human Feedback，基于人类反馈的强化学习），让模型从「能生成语言」变成**「更会按照人的意图回答」**。

2022 年 11 月，ChatGPT 发布。它不是学术上最原创的，但它把几个关键因素一起带进了大众产品：
1. 对话界面非常自然
2. 模型比过去所有聊天机器人都聪明
3. 直接免费体验
4. 「我问一句，它回一句」的交互方式极其低门槛

**普通人第一次大规模感受到：这不是以前那种死板客服机器人。**

> 参考：[InstructGPT 论文](https://proceedings.neurips.cc/paper/2022/hash/b1efde53be364a73914f58805a001731-Abstract.html) | [ChatGPT 发布](https://www.openai.com/blog/chatgpt/)

---

### 2.6  第六阶段：产业生态爆炸——从聊天到干活（2023-2024）

#### 2023：开源权重生态崛起

- **Llama 2**（Meta，2023.7）：开源权重发布，让更多公司和开发者可以自己部署模型。生态从「少数闭源 API」变成「闭源 + 开源并行」。
- **Mistral 7B**（2023.9）：证明了小模型也能有竞争力，降低部署门槛。
- **Gemini**（Google，2023.12）：Google 的多模态模型首次亮相，原生支持文本、图像、音频。

#### 推理基础设施爆发

随着模型开始大规模上线，大家发现：**模型不是不能跑，而是太慢、太贵、显存占用太大。** 行业重心开始从「怎么训练更强模型」转向「怎么更高效地服务模型」。

| 技术 | 年份 | 解决的问题 | 当前状态 |
|------|------|-----------|---------|
| **FlashAttention** | 2022-2023 | 长上下文 Attention 的速度和显存问题——从 GPU 内存访问层面做优化 | v3 已发布，主流模型标配 |
| **vLLM / PagedAttention** | 2023 | KV Cache 浪费显存、碎片化导致 batch size 受限——借鉴操作系统分页思想管理显存 | PyTorch Foundation 项目，400,000+ GPU 部署，已成 K8s 上 LLM 服务标准 |
| **SGLang** | 2024 | 推理框架的编程模型和调度效率，RadixAttention 前缀缓存 | PyTorch Ecosystem 项目，DeepSeek V3 上比 vLLM 快 3.1× |
| **HuggingFace TGI** | 2021-2026 | 早期的自托管 LLM 推理服务 | **2025.12 进入维护模式，2026.3 归档只读**，市场被 vLLM + SGLang 取代 |
| **Ollama** | 2023-2026 | 本地 LLM 推理工具，一键拉取运行开源模型 | 个人开发/本地实验首选，但不支持 Continuous Batching 和分布式部署，不适合生产 |

> 以上各项的 stars 数、版本号等精确数据：vLLM ~86k / SGLang ~30k / Ollama ~174k / TGI 已归档。完整数据见 [ai-engineering.md §5](ai-engineering.md)。

这些技术的出现标志着一个重要转变：**AI 生态正式进入「推理基础设施」时代。**

> 参考：[FlashAttention 论文](https://papers.nips.cc/paper_files/paper/2022/hash/67d57c32e20fd0a7a302cb81d36e40d5-Abstract-Conference.html) | [vLLM / PagedAttention 论文](https://arxiv.org/abs/2309.06180)

#### 产品形态多元化

- **Copilot**（GitHub，2023）：大模型进入编码场景
- **Cursor**（2024）：不只是补全代码，而是「模型 + IDE + 工具调用」的编程代理
- **Claude Code**（Anthropic，2025）：终端原生 AI 编程助手，具备工具调用和自主编码能力的 CLI Agent
- **Claude 3.5 Sonnet**（Anthropic，2024.6）：在编码和推理上表现突出，200K 上下文

---

### 2.7  第七阶段：推理时代与架构革命（2025-2026）

这一阶段还在进行中。核心主题：**从「更大的模型」转向「更聪明地使用模型」。**

#### 2025 上半年：推理能力的涌现

| 时间 | 事件 | 意义 |
|------|------|------|
| 2025.1 | **DeepSeek-R1** 发布 | 用纯强化学习 GRPO（组相对策略优化——不需要单独训练评估模型，通过组内比较自行学习）+ 无人工标注的推理过程，模型涌现出自主推理能力（「aha moment」——模型自己发现错误并修正）。完全开源，震动行业 |
| 2025.4 | **Qwen3** 发布 | 开源 8 个模型（0.6B-235B），统一「思考/非思考」模式，Apache 2.0 协议 |
| 2025.5 | **Claude 4** 系列发布 | Claude Opus 4 / Sonnet 4，200K 上下文，Anthropic 的旗舰推理模型 |
| 2025.6 | **Llama 4** 公布细节 | Meta 首次在 Llama 引入 MoE（混合专家），Scout 模型支持 1000 万 token 上下文 |

## 第三章：深入理解 5 个关键问题

如果你把这 5 个问题真正搞懂，你对 AI 的理解就不再停留在「会聊天的黑盒」层面，而是开始进入「我知道它为什么能跑起来、为什么难、为什么会有这些岗位」的层面。

---

### 3.1  训练和推理的区别是什么

**最短答案：** 训练是把模型「学出来」，推理是把学好的模型「拿来用」。

#### 用普通人能理解的话说

- **训练**：像「上学、做题、练习」——模型接触大量数据，不断调整自己的参数
- **推理**：像「毕业以后正式上岗回答问题」——参数固定，只根据输入生成输出

#### 训练为什么这么贵

训练不只是「算一遍结果」，而是每轮都要做四件事：

1. **前向计算**：先算出预测结果
2. **计算误差**：看与真实答案差多少
3. **反向传播**：把误差信号逐层往回传
4. **更新参数**：让模型下一次更接近正确答案

大模型训练需要：
- 几万亿 token 的训练数据
- 成百上千张 GPU 跑几周到几个月
- 几百万到几千万美元的成本

#### 推理为什么也不便宜

很多人以为「训练才贵，推理很轻」，这不对。大模型推理成本也很高：

1. 模型参数大（权重常驻显存）
2. 上下文可能很长（KV Cache 线性增长）
3. 生成不是一次性输出，而是一个 token 一个 token 地生成（串行）
4. 高并发下 GPU 和显存争抢激烈

所以产业界极其重视「推理成本优化」——每省 10% 的推理成本，在百万用户级别就是巨大的利润。

#### 跟你的关系

你日常接触到的几乎都是推理，不是训练。很多岗位和项目只做推理，不做训练。比如企业内部问答、AI 助手、AI 编码工具——这些更多是在做模型部署、推理优化、请求调度、缓存管理、观测与发布。

那接下来的问题就很自然了：为什么推理能跑这么快？为什么今天所有大模型都用同一个架构？这要回到 2017 年，看 Transformer 是怎么取代 LSTM 的。

---

### 3.2  Transformer 为什么比 LSTM 更适合大模型

#### LSTM 为什么曾经重要

LSTM 解决了普通 RNN「处理长序列时会失忆」的问题。比如一句话：「虽然前面铺垫了很多内容，但最后一个代词'它'到底指什么？」模型需要记住很远之前的信息——LSTM 通过门控机制改善了这个问题。

#### 但 LSTM 不适合今天的大模型

核心原因：**它天生不够适合超大规模并行训练。**

LSTM/RNN 是「顺序执行」的：先处理第 1 个词 → 再处理第 2 个 → 再处理第 3 个。这意味着：
- 难以高效并行
- 训练速度受限于串行瓶颈
- 扩展到超长上下文和超大数据时很吃亏

#### Transformer 的关键优势

**它让序列中各位置可以并行处理。** Attention 机制让当前 token 可以直接「看」上下文中的所有其他 token，不必像 RNN 那样一步一传。

| | LSTM | Transformer |
|---|---|---|
| **训练并行性** | 差（串行依赖） | 好（可并行处理所有位置） |
| **长距离依赖** | 改善但不完美 | 直接建模任意距离关系 |
| **扩展到超大规模** | 困难 | 天然适合 GPU 矩阵运算 |
| **大模型时代的角色** | 已退出主流 | 所有主流模型的底座 |

#### 直觉类比

- **LSTM**：像一圈人按顺序传话——第一个人告诉第二个人，第二个人告诉第三个人……信息是顺着链条传的，传远了容易走样
- **Transformer**：像一个会议室里所有人都能同时看白板、同时关注重点信息——效率高得多

**如果没有 Transformer，GPT、Claude、DeepSeek 这些大模型时代的产品，很可能不会以今天这种形式和速度出现。**

---

### 3.3  RAG 解决了什么、没解决什么

#### 最短定义

**RAG（Retrieval-Augmented Generation，检索增强生成）= 先查，再答。**

#### 解决了什么

**问题 1：模型知识不是最新的。** 模型训练完后参数固定，新发生的事情它不知道。RAG 通过检索外部知识库来补这个问题。

**问题 2：企业私有知识不在通用模型里。** 公司内部 SOP、产品手册、合同模板、历史项目文档——这些不会出现在通用大模型训练数据里。RAG 把这些接进来。

**问题 3：需要出处和证据。** 纯模型回答时难以告诉你「信息来源」。RAG 可以把检索到的文档片段一起呈现，提升可追溯性。

#### 没解决什么

**没解决 1：模型本身推理能力不够。** RAG 只是给模型更多外部信息，不是给模型换脑子。逻辑能力差的模型，给了资料也总结不好。

**没解决 2：检索不准时，回答也会偏。** RAG 效果高度依赖检索质量。文档切分差、向量检索不准、排名错了，模型就基于错误资料回答。

**没解决 3：幻觉不会彻底消失。** 「上了 RAG 就没有幻觉」是误解。RAG 只能降低一部分幻觉——模型仍可能误读文档、过度推断、自己编造。

#### 实用理解

RAG 更像**给一个会说话的模型配了「资料库」**，而不是让它自动变成事实机器。从产品落地角度看，RAG 是企业最现实的切入点之一——不用从零训练模型，可以快速接企业知识，能直接形成问答、助手、搜索增强等产品。

RAG 解决了「模型不知道」的问题。但还有一个更底层的问题，无论你用不用 RAG 都会遇到：**模型生成文本的时候，GPU 显存是怎么被吃掉的？为什么长对话会让系统变慢？** 这就引出了推理系统最核心的瓶颈——KV Cache。

---

### 3.4  为什么 KV Cache 是推理系统的核心瓶颈

#### 先讲生成式模型怎么输出内容

大语言模型不是一次把整段话吐出来。它是这样生成的：

```
看当前输入 → 预测下一个 token → 把新 token 接回输入 → 再预测下一个 → 重复
```

假设已经生成了 500 个 token。如果每生成第 501 个 token 都要把前面 500 个完整重算一遍，代价极高。

#### KV Cache 做什么

**缓存前面已经算过的中间状态，后续生成时直接读取，不重算。** KV Cache = Key-Value Cache，缓存的正是 Attention 计算中每个 token 在各层的 Key 和 Value。

有了 KV Cache，生成第 501 个 token 时只需要：读全量缓存 + 算 1 个新 token 的 Attention → 缓存末尾追加一行。

#### 但它为什么成了「大问题」

因为 KV Cache 非常占显存，且随着以下因素线性增长：

```
KV Cache 大小 ∝ 层数 × 序列长度 × 维度 × 并发请求数
```

- 一个请求长对话几千 token，缓存几十 MB
- 100 个用户同时来，显存直接爆
- 不同请求长度不同，产生显存碎片
- batch 不好做，GPU 利用率不稳定

#### 这为什么会催生推理基础设施

大模型线上服务的真实问题不是「模型能不能回答」，而是：

- 同时 100 个用户来怎么办？
- 长上下文用户会不会把显存吃爆？
- 如何把多个请求合批又不让延迟爆炸？
- 如何减少 KV Cache 浪费？

这就是为什么 vLLM（PagedAttention 借鉴操作系统分页管理 KV Cache）、FlashAttention（优化 GPU 内存访问模式）、DeepSeek 的 MLA/CSA（压缩 K/V）等技术如此重要。

#### 直觉类比

KV Cache 像**写长文时的草稿和记忆区**。如果每写一句话都要把前面所有内容重新从零推导一遍，效率极差。于是系统保留中间记忆——但记忆区很占地方，人一多就不够用，管理不好就混乱。AI 基础设施工程，很多就是在解决这个「记忆区如何高效管理」的问题。

---

### 3.5  一个 AI 产品从请求到回答的完整链路

以你在 Cursor 中输入「帮我解释这个函数，并修复潜在空指针问题」为例：

```text
第一步：用户发请求
  → 前端/客户端接收输入

第二步：服务入口
  → 认证鉴权 → 会话管理 → 限流 → 路由 → 日志记录

第三步：上下文构建（非常关键）
  → 读取当前文件、项目结构、历史对话
  → 拼成模型的输入 Prompt

第四步：是否走 RAG / 检索（可选）
  → 需要查知识库？检索文档片段 → 拼进 Prompt

第五步：是否调工具（可选）
  → 模型判断需要读文件/修改代码/执行命令
  → 系统执行工具调用 → 结果回给模型 → 继续推理

第六步：进入推理引擎（AI Infra 核心区）
  → 分词（Tokenize）
  → Prefill：并行处理全部输入，建好 KV Cache
  → Decode：逐 token 生成，每次追加到 KV Cache
  → 流式推回结果

第七步：结果后处理
  → 格式化、安全过滤、转成代码 patch、结构化 JSON

第八步：返回给用户 + 写入日志/指标
  → 请求耗时、token 数量、错误码、模型版本、工具调用记录
```

**记住这条链路：**

```text
用户请求
  → 鉴权/路由
  → 上下文构建
  → 检索(RAG，可选)
  → 工具调用(可选)
  → 模型推理（Prefill → Decode 循环）
  → 结果后处理
  → 返回用户
  → 日志/指标/追踪
```

这说明一件事：**一个 AI 产品不是「一个模型文件 + 一个聊天框」，而是一个复杂的分布式系统。** 你后面学 AI 工程，真正会接触到的正是这条链路上的某一段或几段。

### 4.1  为什么今天出现大量 AI 基础设施岗位

### 4.2  DeepSeek 意味着什么

DeepSeek 对行业的影响，不只是「又一个强模型」。它让很多人第一次意识到几件事：

#### 正确理解

**GPU 仍然非常重要，但「只堆 GPU」已经不是唯一竞争力。** 真正决定竞争力的还有：

- 训练策略（GRPO 无 critic 模型，RLVR 可验证奖励自学推理）
- 工程优化（MLA → CSA+HCA，KV Cache 极致压缩，推理成本大幅下降）
- 开源策略（MIT 协议，权重全开放，直接刺激了整个开源生态）
- 成本控制（训练成本远低于同期同等能力模型）

#### 为什么大家会产生「GPU 不再是唯一答案」的印象

因为过去对大模型的印象是：谁卡多、谁钱多、谁 GPU 多、谁就强。但 DeepSeek 证明了：架构创新（MLA、MoE）和训练方法创新（GRPO）可以显著缩小资源差距。

#### 但不要误解

不能把它理解成「没 GPU 也无所谓」或「算力时代结束了」。更准确的理解是：

**今天的 AI 竞争，从「单纯拼资源」转向「资源 + 算法 + 工程 + 生态」的综合竞争。**

> 参考：[DeepSeek-R1 技术报告](https://arxiv.org/abs/2501.12948)

---

### 4.3  今天的 AI 生态在争夺什么

今天已经不是单纯争夺「谁有最强模型」。更完整地说，行业在争夺：

1. **谁有更强的基础模型** → 模型能力是起点
2. **谁能更便宜地训练和推理** → 成本决定商业化可行性
3. **谁能更快接入实际场景** → 从模型到产品的时间窗口
4. **谁能构建更强的工具和开发者生态** → 做大蛋糕的能力
5. **谁能更稳定地交付到企业生产环境** → 可靠性和可运维性

所以今天你看到的竞争，不只是模型榜单竞争，而是：模型竞争 × 芯片/算力竞争 × 推理系统竞争 × 产品交互竞争 × 开发生态竞争。

---

### 4.4  你接下来应该学什么

#### 建立核心认知（现在）

**AI 不是「魔法人」**——它是一个极强的模式建模系统，不是真的人类意识。**但它会越来越像「协作对象」**——因为模型本体 + 工具 + 工作流 + 记忆系统组合之后，在很多场景里确实表现得像能协作的人。

**真正让它有用的，不只是模型聪明**，而是：知识怎么接、工具怎么调、延迟怎么控、成本怎么压、故障怎么排、版本怎么发。

也就是说：**未来真正有价值的，不只是「会问 AI」，而是「知道怎样把 AI 接进真实系统里」。**

#### 学习路径

按这个顺序：

1. **读完本文后，接着读 [ai-engineering.md](ai-engineering.md)**——AI 工程实践：LLM 内部机制（Attention/FFN/Prefill/Decode/训练）+ AI Infra 工种 + 各地区岗位市场 + 2026 技术栈数据
2. **再搞懂本文第三章 5 个问题**（训练/推理、Transformer、RAG、KV Cache、完整链路）
3. **再搞懂推理系统**：batch、KV Cache 管理、吞吐 vs 延迟、显存优化
4. **最后看 [ai-infra-project-plan.md](ai-infra-project-plan.md)**——动手做项目（Step 0→4）

#### 你的项目路径选择

两条路径都有效：**Python + HF Transformers**（LLM 应用工程主流，大陆岗位最多）或 **C + ONNX Runtime**（边端/嵌入式/跨平台）。当前计划选择了 Python 路径。

> 两条路径的详细说明（适合岗位、技术栈、对应 Step）见 [ai-engineering.md §6](ai-engineering.md)。
