# FluffyNotes 文档写作规则

## 项目定位

本项目保存个人在工作、技术调研和跨领域学习中形成的文档。文档首先服务于作者自己的理解和复习，也可能公开给具有开发经验、但未必熟悉当前主题的读者。

写作目标是留下可复查、可验证、能帮助读者建立工程理解的资料，不追求覆盖所有知识，也不把文档写成百科、宣传稿或 AI 生成内容的汇编。

## 开始写作前

- 先确认文档要解决的问题、预期读者和使用场景。目标不清楚时先询问，不直接扩写。
- 先查看同目录已有文档，避免重复定义、结论冲突和职责重叠。已有文档可以承载内容时，优先补充原文。
- `database/` 下文档尽量自洽，少用指向其他本地 `.md` 的交叉引用。
- 区分学习笔记、源码解读、实验记录、教程和执行计划。不要把多种任务强行塞进同一种模板。
- 面向“有开发经验、刚接触该领域”的读者。默认读者理解编程、命令行、网络和常见工程概念，不重复讲通用基础；领域特有概念仍需交代背景。

## 内容与结构

- 从实际问题或观察切入，尽快说明本文解决什么。删除泛泛的时代背景和仪式化开场。
- 内容按理解所需的顺序展开，不固定套用“定义 → 分类 → 总结”或三段式结构。
- 解释技术时优先写清机制、输入输出、约束、验证方法和适用边界。类比只能辅助理解，不能把相似写成等价。
- 示例应服务于当前问题。代码和命令必须说明前提、预期结果以及关键限制。
- 教程必须包含可验证的完成标准；实验记录必须区分预期、实际结果和结论；计划必须说明优先级及停止条件。
- 表格只用于确实需要横向比较或快速查询的内容。存在因果或推导关系时使用正文。
- 列表只保留并列项目。能够自然说清的内容不要拆成大量短条目。
- 一个观点完整表达一次。不要在导语、正文、总结和自检中反复改写。

## 事实与来源

- 明确区分事实、推断、经验判断和建议，不把后面三类写成已验证事实。
- 时效性信息、版本、性能数据、市场数据和项目状态优先核对官方文档、源码、论文或原始公告，并标明查询日期。
- GitHub stars 只能说明关注度，不能单独证明技术成熟度、行业采用率或方案适用性。
- 找不到可靠来源时保留疑问或标注待核实，不补造数字、出处、经历和实验结果。
- 引用应尽量指向具体页面或源码位置，避免只写“行业普遍认为”“公开资料显示”。

## 中文表达

- 使用自然、克制的中文技术写作。句子长短可以变化，段落不必保持相同节奏。
- 默认省略不必要的人称。避免频繁使用“你”“我们”“大家”“作为开发者”“用户需要”等代称；需要操作指令时直接写动作。
- 不虚构作者经历、读者反应或对话。需要第一人称时，仅用于记录真实选择、实验和判断。
- 避免连续使用“首先、其次、最后”“值得注意的是”“显而易见”“换句话说”“一句话总结”“关键结论”“真正重要的是”“下面我们来”等组织话术。
- “不是……而是……”只在确有二元纠正关系时使用，不作为默认强调句式。
- 少用空泛判断，如“赋能”“颠覆”“全面提升”“事实标准”“时代已经到来”。重要性应由机制、数据或实际影响体现。
- 不用口号式结尾，不临时拔高主题。内容已经结束时直接结束。
- 控制加粗、引用块、括号、冒号和破折号的密度。加粗只标记确实需要定位的术语或结论。
- 保持术语、品牌大小写、数字单位和引号样式一致。

## 修改与交付

- 润色时保留原有事实、立场和有效细节，优先删除重复和重排结构，不做同义词替换式改写。
- 完稿后依次检查：事实是否可靠、主线是否连续、读者前置知识是否合适、是否存在重复、结论是否强于证据、是否残留模板化表达。
- 如果文档准备公开，再检查标题是否准确、链接是否有效、命令能否执行、示例能否复现，以及私人信息是否已移除。
